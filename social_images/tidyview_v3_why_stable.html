std::HashMap (SipHash)
84–1633 ns · 19.4× swing
!
Random SipHash key per process. Same input lands at different slots in different runs. Cache layout drifts.
!
Adaptive table sizing. Power-of-2 grow events copy the old table; allocation is bursty. Final layout depends on insertion path.
!
Probe sequence is seed-dependent. Robin-hood probing means cache footprint shifts per process.
!
Memory pressure penalizes it harder. Fresh process + cold cache + random layout = the 1633 ns peak.
DHarht Memory (sealed)
105–992 ns · 9.4× swing
✓
Pure splitmix64 scatter. Same input → same shard, same prefix, same slot. fn(u64) → u64 with no state.
✓
Layout fixed at seal(). 256 shards × 16-bit sparse paged front directory. Never reshapes after build.
✓
Direct indexing, no probing. Front directory is a jump table; MicroBucket scan is bounded at 16 entries with parallel match_mask.
✓
Bursty allocator pressure absent. Per-shard slab arenas append monotonically during build, then zero allocation post-seal.

What HashMap structurally cannot offer

unique to DHarht Memory
✓Cross-process byte-equal iter
Two processes, same input → identical iteration order. Use as snapshot identifier or audit checksum.
HashMap: random seed kills this
✓HashDoS-bounded by construction
Worst-case bucket = 16 entries. Beyond that, ART fallback caps depth at 8 byte-radix levels. No probe-length amplification.
HashMap: random seed is the *only* defense
✓shape_hash() determinism proof
One u64 hash over the sorted entry set + counters. Byte-equal across runs proves layout is what you expect.
HashMap: no such surface exists
✓Adversarial diagnostics surface
overflow_count() and max_collision_group() are queryable. Detect bucket pressure before it becomes a problem.
HashMap: no equivalent counters
✓Pre-computed hash lookup
lookup_by_hash(u64) accepts an externally-computed content hash. Skip rehashing in pipelines that already carry it.
HashMap: no API to pass hash directly
✓Predictable SLA
9.4× variance vs HashMap's 19.4×. When p99 latency matters more than p50, the tighter envelope is the win.
HashMap: tail latency is harder to bound
