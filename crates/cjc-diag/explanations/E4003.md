# A nondeterministic operation appeared in a deterministic context

## What happened

Inside a function declared `/det` (deterministic) -- or reachable
from one -- the compiler detected an operation whose result depends
on something other than the function's arguments. Examples:

- `random()` without an explicit RNG argument
- `current_time()`, `now()`, anything reading the clock
- `HashMap` iteration (random order)
- Reading process environment variables
- Thread-scheduling-dependent timing reads

## Why it matters

Determinism is one of CJC-Lang's load-bearing properties. The whole
toolchain -- Kahan accumulators, BTreeMap iteration, SplitMix64
seeded RNG, no FMA, parity-tested executors -- exists to make
"same seed, same output" hold across runs and platforms.

A nondeterministic operation breaks that guarantee. CJC-Lang's effect
system catches this at compile time: functions can declare they need
nondeterminism (`/nondet`), and the compiler refuses to call them
from a `/det` context.

This is what lets CJC-Lang make its audit-reproducibility claim:
every function in a `/det` chain is statically proven to be a
function of its inputs alone.

## How to fix it

**Thread the source of nondeterminism in explicitly:**

```cjcl
// before -- implicit RNG access
fn sample(n: i64) -> [f64] /det {
    let mut out = [];
    for _ in 0..n {
        out = array_push(out, random());    // E4003
    }
    out
}

// fix -- accept the RNG state as an argument
fn sample(rng: Rng, n: i64) -> [f64] /det {
    let mut out = [];
    for _ in 0..n {
        out = array_push(out, rng.next_f64());
    }
    out
}
```

Now the function is deterministic *given the same `rng` value*.
That's the strongest determinism property you can have on RNG-using code.

**Or declare the function /nondet:**

```cjcl
fn current_temp() -> f64 /nondet {
    sensor_read()    // genuinely depends on external state
}
```

Callers see `/nondet` in the signature and know they're crossing the
deterministic boundary.

## Common pitfall

Iterating a `HashMap` is nondeterministic in many languages. CJC-Lang
mostly uses `BTreeMap` (deterministic by key order) precisely to
avoid this. If you see E4003 on what looks like a normal map iteration,
check whether the map type is the deterministic one.
