# Return type doesn't match the function's declared return type

## What happened

A function's body produces a value whose type doesn't match the
return type declared in its signature. The diagnostic shows both: the
declared type and the type the body actually produces.

```cjcl
fn solve(x: i64) -> i64 {
    "not a number"     // E2011 -- declared i64, produces String
}
```

## Why it matters

The return type in the signature is the function's **contract** with
every caller. Callers wire downstream code based on the declared
return type -- if the body silently produces something else, every
caller becomes wrong.

CJC-Lang checks this contract at function definition time, not at
call time, so the error surfaces locally instead of as a cascade of
type mismatches at every call site.

## How to fix it

**Make the body produce the declared type:**

```cjcl
fn solve(x: i64) -> i64 {
    x * 2     // produces i64, matches declared
}
```

**Or change the declared type if the body is correct:**

```cjcl
fn solve(x: i64) -> String {
    "not a number"     // declared changed to match body
}
```

**Trailing semicolons can flip the return type to Unit:**

```cjcl
fn solve(x: i64) -> i64 {
    x * 2;     // semicolon turns it into a statement -- body returns Unit
}    // E2011 -- declared i64, body produces Unit

// fix -- drop the semicolon
fn solve(x: i64) -> i64 {
    x * 2
}
```

**Branches must all agree with the declared type:**

```cjcl
fn classify(x: i64) -> String {
    if x > 0 {
        "positive"
    } else {
        0          // E2011 here -- declared String, this branch is i64
    }
}

// fix
fn classify(x: i64) -> String {
    if x > 0 { "positive" } else { "non-positive" }
}
```

## Common pitfall

The semicolon trap (turning the tail expression into a statement) is
the most common cause. CJC-Lang uses the **absence** of a semicolon on
the last expression to signal "this is the return value." If you add a
`;`, the function effectively returns `Unit` and the declared type
becomes mismatched.
