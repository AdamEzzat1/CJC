# Invalid byte literal

## What happened

The lexer was scanning a byte literal (`b"..."` for byte strings, or
`b'X'` for single bytes) and found content that isn't representable as
raw bytes. Common causes:

- A `\u{HHHH}` escape inside a byte literal (Unicode escapes produce
  multi-byte sequences; byte literals are exactly one byte per slot)
- A non-ASCII character without an escape (byte literals are ASCII only)
- A malformed `\x{HH}` escape (must be exactly two hex digits)
- A multi-character byte literal `b'ab'` (byte chars are single byte)

## Why it matters

Byte literals are explicitly NOT Unicode. Every position in a
`b"..."` is one byte (`u8`); every `b'X'` is one byte. The lexer
enforces this so the type of the literal is unambiguous at the source.

If CJC-Lang allowed `\u{...}` in byte literals, the literal's length
would depend on UTF-8 encoding rules -- `b"\u{1F600}"` would be FOUR
bytes long (the UTF-8 encoding of the emoji), which is rarely what
someone writing a byte literal intends. Rejecting the escape
clarifies the contract.

## How to fix it

**Unicode escape in byte literal -- use \\x or a regular string:**

```cjcl
let emoji = b"\u{1F600}";    // E0010

// fix option A -- use a regular string for Unicode
let emoji: String = "\u{1F600}";

// fix option B -- spell out the UTF-8 bytes if you really want bytes
let emoji = b"\xF0\x9F\x98\x80";
```

**Non-ASCII char without escape:**

```cjcl
let cafe = b"café";    // E0010 -- `é` is non-ASCII

// fix
let cafe = b"caf\xC3\xA9";   // UTF-8 bytes of `é`
// or just use a String:
let cafe = "café";
```

**Malformed \\x escape:**

```cjcl
let one = b"\x1";      // E0010 -- need exactly two hex digits

// fix
let one = b"\x01";
```

**Multi-byte char literal:**

```cjcl
let c = b'ab';         // E0010 -- byte char literals are single byte

// fix -- use a byte string for two bytes
let c = b"ab";
```

## Common pitfall

If your goal is "raw bytes for binary I/O" but you're hitting E0010
constantly, the language usually wants you to use a regular `String`
plus `string.bytes()` to get the bytes at use-site. Byte literals are
for fixed-size byte sequences known at compile time (magic numbers,
fixed protocol fragments) -- not a general substitute for text.
