# Invalid pattern in a match arm

## What happened

The parser was reading a `match` arm and found something that can't
serve as a pattern. CJC-Lang's match patterns are limited to:

- Literals (`0`, `"foo"`, `true`)
- Variable bindings (`x`, `name`)
- Wildcards (`_`)
- Tuple patterns (`(a, b)`)
- Struct patterns (`Point { x, y }`)
- Or-patterns (`1 | 2 | 3`)
- Guarded patterns (`x if x > 0`)

Anything outside this set -- an arbitrary expression, a function call,
a binary operation -- is rejected because match patterns are a
distinct grammar from expressions.

## Why it matters

Pattern matching is the language's primary tool for case analysis, and
CJC-Lang requires patterns to be **structurally inspectable at compile
time**. The compiler proves match exhaustiveness, drives MIR lowering
into a decision tree, and checks reachability for each arm -- all of
which require patterns to be a closed, simple grammar.

If patterns could be arbitrary expressions, exhaustiveness analysis
would become undecidable in general, and the optimizer couldn't lower
matches into efficient dispatch.

## How to fix it

**Compute the value first, then match on a binding:**

```cjcl
match compute_score() + 10 {   // E1005 -- not a pattern
    50 => "high",
    _ => "other",
}

// fix
let total = compute_score() + 10;
match total {
    50 => "high",
    _ => "other",
}
```

**Replace a function call in a pattern with a guard:**

```cjcl
match x {
    is_prime(x) => "prime",   // E1005 -- is_prime() is a call, not a pattern
    _ => "composite",
}

// fix -- use a guard
match x {
    x if is_prime(x) => "prime",
    _ => "composite",
}
```

## Common pitfall

The right-hand side of `=>` IS an expression (you can call functions,
do arithmetic, etc. there). It's only the left-hand side -- the
pattern itself -- that's restricted.
