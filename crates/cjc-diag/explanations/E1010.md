# Missing return type after `->`

## What happened

You wrote `->` in a function signature (the arrow that introduces the
return type) but didn't follow it with a type:

```cjcl
fn solve(x: i64) -> { x * 2 }   // E1010 -- type missing after `->`
```

## Why it matters

In CJC-Lang, the return type is part of the function's **contract** --
every caller sees it as the function's promise. Even functions that
return "nothing" must say so explicitly: the type is `Unit` (or
`Void`, depending on the language version).

If `->` were optional or could implicitly default to something, the
"contract at the signature" property would break -- callers would
have to read the body to know what the function returns. That's
exactly the situation CJC-Lang avoids.

## How to fix it

Add the return type:

```cjcl
// returns i64
fn solve(x: i64) -> i64 { x * 2 }

// returns nothing meaningful (Unit / Void)
fn log(msg: String) -> Unit {
    print(msg);
}

// or just omit the `->` for functions that return Unit
fn log(msg: String) {
    print(msg);
}
```

The last form (omitting both `->` and the type) is the conventional
way to write a Unit-returning function. The `-> Unit` is implied by
the absence of `->`.

## Common pitfall

`fn solve(x: i64) -> () { ... }` would NOT be valid CJC-Lang. The
empty tuple `()` is a value, but the language's "no return value" type
is named `Unit` (not `()`). Use the no-arrow form or write `Unit`
explicitly.
