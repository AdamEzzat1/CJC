# Invalid declaration at top level

## What happened

The parser was at the top level of a file (the outermost scope, not
inside any function or block) and found something that can't appear
there. Valid top-level forms are:

- `fn ...` -- function declarations
- `struct ...` / `record ...` / `class ...` -- type declarations
- `trait ...` / `impl ...` -- traits and impls
- `import ...` -- module imports
- `mod ...` -- module declarations

Statements like `let`, `if`, expressions, etc. cannot appear at the
top level; they must be inside a function body.

## Why it matters

CJC-Lang doesn't have "script mode" execution where top-level
statements run sequentially. The entry point of any program is the
`main` function. This is a deliberate choice -- it makes the program's
execution start a single, predictable point, which matters for
determinism (every run of the program does exactly the same sequence
of things in the same order).

A script-style language where any statement can run at top level has
ambiguity about *order* (especially across multiple files), which is
the kind of nondeterminism CJC-Lang refuses.

## How to fix it

Move executable statements into `main` (or another function):

```cjcl
// E1011 at top level
let result = compute();
print(result);

// fix
fn main() -> i64 {
    let result = compute();
    print(result);
    0
}
```

For one-shot computation that should run when the program starts,
`main` is the place. Initialization that needs to happen before main
doesn't exist in CJC-Lang -- there's no equivalent of C++'s static
constructors or Python's module-level code.

## Common pitfall

If you're migrating a Python/JavaScript file with top-level code, the
mechanical conversion is to wrap everything in `fn main() -> i64 { ...
0 }`. Definitions (functions, structs, imports) stay at top level;
executable statements go inside main.
