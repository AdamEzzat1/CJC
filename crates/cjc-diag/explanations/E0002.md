# A string literal was never closed

## What happened

The lexer found an opening `"` (or `'` for char literals) but reached
the end of the line, or the end of the file, before finding the
matching closing quote. Everything between the unclosed quote and the
file end was being treated as string content, which usually causes a
cascade of errors after the first.

## Why it matters

Strings in CJC-Lang are delimited by paired quotes -- the language
doesn't have heredocs, triple-quoted strings, or other long-form
literal forms (yet). A missing close quote isn't a recoverable parse
error; it changes the boundary of every token that follows.

The lexer reports the location where the quote opened, not where it
should have closed, because the closer is by definition missing -- the
lexer has no way to guess where you meant it to go.

## How to fix it

**Missing close quote:**

```cjcl
let greeting = "hello;   // E0002 -- string never closes

// fix
let greeting = "hello";
```

**Newline mid-string:**

CJC-Lang strings are **single-line by default**. Embedded newlines must
be escaped:

```cjcl
let banner = "line one
              line two";    // E0002 -- string literal broke on newline

// fix
let banner = "line one\nline two";
```

**Escape gone wrong:**

A backslash before the closing quote escapes the quote, so the lexer
keeps scanning:

```cjcl
let path = "C:\\Users\";    // E0002 -- the `\"` escaped the closing quote

// fix
let path = "C:\\Users\\";   // double-backslash to mean literal backslash
```

## Common pitfall

If you see E0002 followed by a flood of other errors, fix E0002 first.
Once the string boundary is correct, the cascading errors usually
disappear. The lexer cannot tell that your "string" actually ended ten
lines ago and now it's parsing what should be regular code as if it
were string content.
