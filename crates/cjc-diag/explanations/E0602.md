# Snapshot type hash mismatch

## What happened

You tried to load a `cjc-snap` snapshot, and the hash of a type stored
in the snapshot doesn't match the hash of the same-named type in the
current program. The stored type's *shape* has changed in a way the
loader cannot bridge.

Examples of type-shape changes that trigger this:

- A struct gained or lost a field
- A field's type changed (e.g., `count: i64` -> `count: f64`)
- An enum variant was added or removed
- Field reordering (if the layout depends on order)

## Why it matters

The type hash is a one-way function of the type's structural
definition. It catches schema evolution that would silently
mis-interpret stored bytes. Without this check, loading a snapshot
saved with `struct Point { x: i64, y: i64 }` into a program that
declares `struct Point { x: f64, y: i64 }` would happily produce
garbage `f64` values from the original `i64` bit patterns.

CJC-Lang's choice to fail loudly here is part of the broader audit
story: a snapshot's contents are only valid if the schema matches the
program loading them.

## How to fix it

**Re-save the snapshot with the new type shape.** This is the
standard fix when you've intentionally changed the type:

```bash
cjcl run --save-snapshot fresh.bin updated_program.cjcl
```

**Or use the old type definition** if the change was unintentional --
roll back the field that drifted.

**For schema-evolution scenarios** (you genuinely need to load old
data into a new program), CJC-Lang doesn't currently have automatic
migration. Write a conversion step that loads the old snapshot using
the old type definitions, transforms the values, and saves them with
the new types.

## Common pitfall

Field reordering counts as a type change for hash purposes, even if
all the field names and types are the same. `struct A { x: i64, y:
i64 }` and `struct A { y: i64, x: i64 }` have different hashes. If
you want a stable wire format, order fields in canonical (usually
alphabetical or declaration) order and don't shuffle them.
