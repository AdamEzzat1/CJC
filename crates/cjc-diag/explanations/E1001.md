# Expected a specific token, found something else

## What happened

The parser was in the middle of parsing a known construct -- a function
signature, a struct definition, a match arm -- and it knew *exactly*
what token should come next (a colon, a comma, a closing brace), but
the source had something different there. The diagnostic shows both:
what was expected and what was found.

```
expected `:`, found `)`
expected `,`, found `if`
expected `}`, found EOF
```

## Why it matters

When the parser fires E1001 instead of the more generic `E1000`, it
means the parser has *high confidence* about what your code was trying
to express. This is good news for you -- the fix is usually exactly
what the message says: insert (or replace with) the expected token at
that position.

The reason CJC-Lang draws this distinction is that **specific** parse
errors are much faster to act on than generic ones. "Unexpected token"
makes you re-read the whole line; "expected `:`, found `)`" tells you
exactly which character to change.

## How to fix it

Almost always: insert (or replace with) the expected token where the
diagnostic points.

**Missing type colon in a function parameter:**

```cjcl
fn solve(x) -> i64 { x * 2 }
//        ^ expected `:` here

// fix
fn solve(x: i64) -> i64 { x * 2 }
```

**Missing comma in an array literal:**

```cjcl
let arr: [i64] = [1 2 3];
//                   ^ expected `,` here

// fix
let arr: [i64] = [1, 2, 3];
```

**Unclosed brace running into EOF:**

```cjcl
fn main() -> i64 {
    let x = 1;
// expected `}`, found EOF

// fix
fn main() -> i64 {
    let x = 1;
    x
}
```

## Common pitfall

When the "found" token is **EOF**, the actual problem is usually
upstream -- a `{` opened a block that never got its matching `}`.
Look for the earliest unbalanced opener; that's where to start.
The reported span is where the parser ran out of source, not where
the imbalance began.

When the "expected" token is `:`, double-check whether you're inside:

- A function signature (parameters need types: `x: i64`)
- A struct field declaration (`field_name: i64`)
- A type ascription (`let x: i64 = ...`)
- A match arm (no colon -- uses `=>`)

E1001 in a match arm usually means you wrote `pat: result` instead of
`pat => result`.
