# Type assertion failed at runtime

## What happened

A runtime type assertion -- typically a downcast from `Any` to a
concrete type, or an unchecked tag-check on a value -- found that the
value's runtime type doesn't match what the code expected. The
diagnostic shows the expected type and the actual type.

```cjcl
fn use_int(value: Any) {
    let x: i64 = value as i64;   // E8005 if `value` is a String
}
```

## Why it matters

Most of CJC-Lang's type system is static -- the type checker proves
type correctness before the program runs, and you never see runtime
type errors. But two paths bypass static checking:

- **`Any` values** -- when you opt into dynamic typing, downcasts back
  to a concrete type must be checked at runtime
- **`unsafe` cast operations** (where present) -- bypassing the static
  check requires the runtime check to fire if the cast is wrong

E8005 is the runtime side of this. It halts deterministically rather
than letting wrong-typed data flow into a kernel that would mis-
interpret it (which would produce garbage results without a clear
signal).

## How to fix it

**Check the runtime type before casting:**

```cjcl
fn use_int(value: Any) {
    if value is i64 {     // or whatever the type-test syntax is
        let x: i64 = value as i64;
        // ...use x...
    } else {
        // handle the other case
    }
}
```

**Avoid `Any` where possible:**

If you're using `Any` because you migrated from a dynamically-typed
language, consider whether your function genuinely needs polymorphism
or whether you can specialize it for the concrete types you actually
use. Specialized versions skip the runtime check entirely.

**For values from JSON/snap deserialization**, validate before use:

```cjcl
let value = snap_load(buf);   // returns Any
let count = match value {
    Value::Int(i) => i,
    _ => 0,                    // explicit fallback rather than failing cast
};
```

## Common pitfall

The `as` operator in CJC-Lang has two flavours: **lossless numeric
conversions** (like `i64 as f64`, which always succeeds) and **type
assertions** on `Any` (which can fail with E8005). Read the cast
direction carefully -- only the latter triggers this error.
