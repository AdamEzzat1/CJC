# A match expression has no arms

## What happened

You wrote `match scrutinee { }` -- a match expression with no arms.
The parser flagged this because an empty match is unreachable code:
it can never produce a value, but it claims to be an expression.

## Why it matters

In CJC-Lang, `match` is an **expression** -- it must produce a value
of some type. To produce a value, at least one arm must be present
(and reachable). A match with zero arms is structurally ill-formed.

Other languages handle this differently:
- Rust allows empty matches on uninhabited types (`!`, `Never`) because
  the compiler can prove they're unreachable; CJC-Lang doesn't yet
  have an uninhabited-type story
- Python (`match`/`case`) requires at least a default `case _` to be
  exhaustive
- Switch statements in C-family languages allow empty bodies, but
  they're statements (not expressions) so the issue doesn't arise

CJC-Lang's choice is the simplest: every match must have at least one
arm, and the compiler also checks that the arms cover all cases (see
`E2009` for non-exhaustive matches).

## How to fix it

If you're stubbing out a match for later, add a wildcard arm:

```cjcl
match x { }            // E1012

// fix -- catch-all so the match is well-formed
match x {
    _ => default_value,
}
```

If you have actual cases in mind, write them:

```cjcl
match x {
    0 => "zero",
    n if n > 0 => "positive",
    _ => "negative",
}
```

## Common pitfall

An empty match isn't the same as a match where every arm panics or
returns -- that's still arms, just with side-effecting bodies. The
parser doesn't check arm bodies for "useful" content; it only requires
the syntactic presence of at least one arm.
