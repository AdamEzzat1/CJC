# Undefined field on a type

## What happened

You accessed `value.field_name` but the type of `value` doesn't have
a field by that name. The diagnostic shows both the field name and
the type, so the typo or wrong-type-assumption is visible.

```cjcl
struct Point { x: i64, y: i64 }

let p = Point { x: 1, y: 2 };
print(p.z);    // E2008 -- no field `z` on Point
```

## Why it matters

CJC-Lang resolves field accesses **at compile time**, not via runtime
attribute lookup like Python or JavaScript. The compiler must know
the struct's layout to lay it out in memory and pick the right offset
for each access. An undefined field can't be looked up, so it's a
compile error.

This is also why CJC-Lang's structs are nominal: the compiler can't
treat "any struct with x and y fields" as "Point" -- the struct type
itself matters. (Anonymous tuples are different; see field access for
those.)

## How to fix it

**Typo:**

```cjcl
struct Vector3 { x: f64, y: f64, z: f64 }

let v = Vector3 { x: 1.0, y: 2.0, z: 3.0 };
let mag = v.z + v.zz;   // E2008 on `zz` -- typo

// fix
let mag = v.z + v.x;
```

**Wrong type assumption:**

```cjcl
struct Point2D { x: f64, y: f64 }
struct Point3D { x: f64, y: f64, z: f64 }

fn use_3d(p: Point2D) {
    print(p.z);    // E2008 -- Point2D has no `z`
}

// fix -- accept the right type
fn use_3d(p: Point3D) {
    print(p.z);
}
```

**Tensor "field" that isn't:**

```cjcl
let t: Tensor[f64] = ...;
print(t.size);    // E2008 -- tensor has `shape()` and `len()`, not `size`

// fix
print(t.len());
```

## Common pitfall

Coming from Python's "duck typing"? CJC-Lang doesn't have duck typing
at the field-access level. If your function should work on multiple
struct types with the same field, you currently have two options:
write separate functions, or use `Any` and a runtime check (paying
the dynamic dispatch cost).
