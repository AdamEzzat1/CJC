# Duplicate type definition

## What happened

You declared two types (structs, enums, records) with the same name in
the same scope. The compiler can't choose between them -- which one
does `Point { x: 1, y: 2 }` refer to?

```cjcl
struct Point { x: i64, y: i64 }
struct Point { x: f64, y: f64 }    // E2014 -- name already taken
```

## Why it matters

Names in CJC-Lang resolve at compile time, deterministically. If two
types could share a name, the compiler would have to make an arbitrary
choice -- often "last declared wins" or "first declared wins" -- and
that choice would become a hidden assumption that breaks when code is
reordered.

CJC-Lang rejects the duplication so you decide explicitly which type
you meant.

## How to fix it

**Rename one of them:**

```cjcl
struct Point2D { x: i64, y: i64 }
struct Point3D { x: f64, y: f64, z: f64 }
```

**Or unify them if they should be the same type:**

```cjcl
struct Point { x: f64, y: f64 }   // pick one set of fields
```

**Move one into a module if they belong to different namespaces:**

```cjcl
// in module geometry/
struct Point { x: f64, y: f64 }

// in module color/
struct Point { x: u8, y: u8 }      // separate namespace, no collision
```

## Common pitfall

Forward declarations and "main + helper" files: if you copy a struct
into a header file and forget to delete the original, E2014 fires.
CJC-Lang doesn't have header files (the module system handles cross-
file sharing), so this pattern usually doesn't apply.
