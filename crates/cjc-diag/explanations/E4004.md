# Allocation in a nogc function

## What happened

Inside a function marked `#[nogc]`, the compiler detected an operation
that would allocate from the garbage-collected heap. This is the more
specific cousin of `E4001`: where `E4001` flags any GC-using operation
(including, say, `gc_alloc` builtin calls), `E4004` specifically flags
*allocations* -- operations that produce a new owned value.

## Why it matters

NoGC functions are CJC-Lang's hardest guarantee about timing and memory
behaviour. The verifier proves -- not "tries to ensure," but *proves*
-- that the marked code does not allocate. This is essential for:

- Inner loops where GC pause variance would dominate the runtime
- Tight kernels (matmul, autodiff backward, ABNG per-row updates)
- Code with timing requirements (real-time-ish use cases)

An allocation slipping into NoGC code defeats the entire guarantee --
which is why the verifier is conservative and the error is non-negotiable.

## How to fix it

**Pre-allocate, mutate in place:**

```cjcl
// before -- E4004 on the fresh array literal
fn fill(n: i64) -> [f64] /nogc {
    let out = [];
    for i in 0..n {
        out = array_push(out, i as f64);    // both lines allocate
    }
    out
}

// fix -- take a buffer in
fn fill(buf: [f64], n: i64) /nogc {
    for i in 0..n {
        buf[i] = i as f64;     // in-place
    }
}
```

**Use NoGC-safe equivalents:**

| Allocating (rejected by /nogc) | In-place equivalent |
|---|---|
| `array_push(arr, v)` | mutate a pre-sized buffer + length counter |
| `format!("...")` | pre-formatted static strings |
| array/tensor literal `[...]` | pre-allocated scratch, fill it |
| string concatenation | pre-allocated string buffer |

**Push the NoGC boundary outward** if the inner code genuinely needs
to allocate -- it's better to have a non-NoGC body called from a
NoGC-marked caller (if that arrangement matches your perf goal) than
to fight the verifier.

## Common pitfall

This and `E4001` overlap heavily; `E4001` is the umbrella "any GC
operation," `E4004` is specifically "an allocation operation." Some
versions of CJC-Lang collapse these into a single code -- in that
case, treat them as the same diagnostic with the same fix recipes.
