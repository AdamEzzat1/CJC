# Missing effect annotation after `/`

## What happened

You wrote a function signature with a `/` (the effect annotation
delimiter) but didn't follow it with a valid effect name. CJC-Lang's
effect annotations look like:

```cjcl
fn solve(x: f64) -> f64 /nogc { ... }    // declares the function is NoGC
fn io_op(path: String) -> String /io { ... }  // declares IO effect
fn random() -> f64 /nondet { ... }       // declares non-deterministic
```

The parser saw `/` but the next token wasn't a recognized effect name,
or was missing entirely.

## Why it matters

Effects in CJC-Lang are declared up-front in the signature so the
language can enforce them statically. A function marked `/nogc` cannot
allocate from the GC; a function with no `/io` effect cannot perform
side effects beyond what its arguments allow; etc. This is core to
CJC-Lang's determinism + audit story -- effects in the signature mean
the compiler can prove what a function CAN and CANNOT do.

The `/` syntax is intentionally lightweight (one character) because
effects appear frequently in numerical/scientific code. But the
grammar requires the effect name to follow immediately, with no
intervening tokens.

## How to fix it

Add the effect name:

```cjcl
// before -- E1013, `/` with nothing after
fn solve(x: f64) -> f64 / { ... }

// fix -- specific effect
fn solve(x: f64) -> f64 /nogc { ... }
```

If you don't actually need an effect annotation, delete the `/`:

```cjcl
fn solve(x: f64) -> f64 { ... }   // no effect annotation = default
```

Multiple effects are comma-separated:

```cjcl
fn step(x: f64) -> f64 /nogc,nondet { ... }
```

## Common pitfall

Effect names are case-sensitive lowercase. `/NoGC` or `/Nogc` will
fire E1013 (the `/` is found, but the next token isn't a recognized
effect identifier). Use `/nogc`.
