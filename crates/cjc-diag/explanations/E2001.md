# A value's type doesn't match what was expected

## What happened

Somewhere in your code, an expression produced a value of one type but the
surrounding context required a different type. The diagnostic shows the
two types that didn't line up — for example:

```
expected `i64`, found `f64`
```

This is usually printed under a caret pointing at the exact expression
that produced the wrong type.

## Why it matters

CJC-Lang doesn't perform implicit numeric coercion. `1 + 1.0` is **not** a
valid expression — `1` is `i64`, `1.0` is `f64`, and the language refuses
to silently widen one into the other.

This is on purpose. Implicit coercion is the source of nearly every subtle
numerical bug in scientific computing:

- Floats and integers round differently. Summing `i64` values and then
  dividing produces different results than treating them as floats from
  the start.
- Mixing `f32` and `f64` quietly downcasts intermediates, which is exactly
  how "the test passes on my machine but fails on the CI runner" happens.
- Tensor operations that silently broadcast across dtypes hide bugs in the
  most performance-critical paths.

By requiring explicit conversions, CJC-Lang forces every type transition
to be a deliberate choice. The trade is: a little extra typing, in
exchange for *every* arithmetic operation having a known, fixed
numerical behaviour.

## How to fix it

The fix depends on what you actually meant.

**If you wanted a float computation, use float literals:**

```cjcl
// before — mixing types
let x: f64 = 1 + 0.5;        // E2001: `1` is i64, not f64

// after — both operands are f64
let x: f64 = 1.0 + 0.5;
```

**If you have an `i64` and need to use it as a float, convert explicitly:**

```cjcl
let count: i64 = 100;
let mean: f64 = total / (count as f64);
```

**If a function call is mismatched, check the signature, not the call:**

```cjcl
fn solve(x: i64) -> i64 { ... }

solve(3.14);   // E2001 — 3.14 is f64, but solve takes i64

// fix option A — truncate at the call site
solve(3.14 as i64);

// fix option B — change the signature if the function should accept floats
fn solve(x: f64) -> f64 { ... }
```

## Common pitfall

**Tensor element types must match too.** Adding two tensors of different
dtypes is an error, not a coerced operation:

```cjcl
let a: Tensor[f64] = ...;
let b: Tensor[f32] = ...;
let c = a + b;   // E2001 — pick one dtype for the whole computation
```

The right answer is almost always to pick one float precision at the top
of your computation and use it everywhere. If you really need to convert,
use `tensor.cast(f64)` so the conversion is visible in the source.
