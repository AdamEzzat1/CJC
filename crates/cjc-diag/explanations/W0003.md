# Code after this point cannot be reached

## What happened

The compiler determined that the marked code can never execute. Common
causes:

- A `return` statement that always fires, followed by more code
- An `if true { ... }` or `if false { ... }` whose dead branch contains statements
- A `match` whose arms collectively cover all possibilities, followed
  by a "fallthrough"
- A loop with no exit path, followed by code after the loop

## Why it matters

Most unreachable code is a bug -- you wrote it intending it to run,
but a structural condition (an early return, a tautological guard)
means it can't. Compilation continues (this is a warning, not an
error), but the dead code will not be in the running program.

The cousin warning `W0001` (unused variable) is about *bindings* the
program doesn't read; `W0003` is about *statements* the program doesn't
execute. They often co-occur because dead code often contains dead
bindings.

The reason CJC-Lang warns rather than errors: occasionally, unreachable
code is intentional -- a placeholder, a debug branch, a defensive
`panic("should not happen")` after an exhaustive `match`. Warning lets
you keep moving while the analyser surfaces the issue.

## How to fix it

**Delete it if you don't need it:**

```cjcl
fn solve(x: i64) -> i64 {
    return x;
    x * 2          // W0003: unreachable
}

// fix
fn solve(x: i64) -> i64 {
    x
}
```

**Restructure if you DO need it:**

```cjcl
fn process(x: i64) -> i64 {
    if x < 0 {
        return -1;
    }
    return x;          // always fires
    log("processed");  // W0003: unreachable
}

// fix -- move the log before the return
fn process(x: i64) -> i64 {
    if x < 0 {
        return -1;
    }
    log("processed");
    x
}
```

## Common pitfall

A `match` that's exhaustive followed by an additional case:

```cjcl
fn classify(x: i64) -> i64 {
    match x {
        0 => 0,
        _ => 1,
    }
    0    // W0003 -- the match returns; this never runs
}
```

Either the match's tail value IS the function's value (delete the
trailing `0`), or you wanted a different return value (move it up
before the match, or restructure the match to choose between paths
that reach the trailing code).

## Note on severity

This is a warning, not an error -- compilation continues, your program
runs, your tests pass. CJC-Lang doesn't currently treat warnings as
fatal by default, but tooling that audits "production-ready" code (the
forthcoming `cjcl ci` strict mode) will eventually surface persistent
warnings. Cleaning them up is cheap; deferring them adds up.
