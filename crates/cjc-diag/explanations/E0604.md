# Type is not snap-compatible

## What happened

You tried to serialize (or deserialize) a value whose type isn't
supported by `cjc-snap`. Some types are intentionally not serializable:

- **Function values** (no portable representation)
- **Closures with non-snap-compatible captures**
- **Foreign-function handles** (e.g., raw file descriptors)
- **Internal compiler types** (HIR nodes, MIR nodes)
- **Types containing any of the above** transitively

The diagnostic identifies the offending type.

## Why it matters

`cjc-snap` is a byte-exact, audit-safe serialization format. Types
that can't be reduced to a stable byte representation can't be
serialized -- and quietly skipping or null-filling them would
silently change the loaded program's behaviour.

CJC-Lang refuses to serialize non-snap-compatible types rather than
serializing them as null or skipping them. The error makes you
decide what the type's representation should be.

## How to fix it

**Replace the non-serializable field with a serializable one:**

```cjcl
// before -- closure is not snap-compatible
struct Handler {
    name: String,
    callback: fn(i64) -> i64,    // E0604 on Handler
}

// fix -- store an identifier that names the callback
struct Handler {
    name: String,
    callback_name: String,        // serializable
}
```

**Or exclude the field from serialization** if your version supports
field-level exclusion attributes:

```cjcl
struct Handler {
    name: String,
    #[snap(skip)]
    callback: fn(i64) -> i64,
}
```

**Or keep the type out of snapshot paths** -- some types are
legitimately unsuitable for serialization (FFI handles, runtime
counters) and should stay in-memory only.

## Common pitfall

If a top-level struct is snap-compatible but its fields aren't, the
error fires at the top level. Trace the error message to find which
nested field is the culprit -- it's usually a closure or a function
pointer buried a level or two deep.
