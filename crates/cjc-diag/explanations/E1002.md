# Expected an expression here

## What happened

The parser reached a position where the grammar requires an *expression*
-- the right-hand side of `=`, the condition of `if`, an argument to a
function call, the value of a `let` binding -- but found something that
can't start an expression. The diagnostic often comes with a hint about
what kind of expression the parser was hoping to see.

## Why it matters

CJC-Lang is **expression-oriented**. Most things in the language *are*
expressions:

- `if cond { a } else { b }` -- produces a value
- `match x { ... }` -- produces a value
- `{ stmt; stmt; tail }` -- produces the value of the tail
- `let x = ...` -- needs an expression on the right of `=`

When the parser says "expected expression," it's saying: this position
is one where the language demands something that produces a value, and
the next token can't possibly start one. Treating expression positions
consistently is part of what makes CJC-Lang's semantics predictable --
every value-producing context uses the same grammar.

## How to fix it

The right fix depends on what's at the cursor:

**Trailing operator without right operand:**

```cjcl
let x = 1 +;          // E1002 on `;`

// fix -- supply the right operand
let x = 1 + 2;
```

**Empty branch in a value-producing `if`:**

```cjcl
let x: i64 = if cond {
    // nothing
} else {              // E1002 inside the empty `{}`
    42
};

// fix -- supply a value (or a default)
let x: i64 = if cond {
    0
} else {
    42
};
```

**`return` used in expression position:**

```cjcl
let x = return 1;    // E1002 -- return is a statement, not an expression

// fix -- in CJC-Lang, the function's tail expression IS the return
fn solve() -> i64 {
    1
}
```

## Common pitfall

In CJC-Lang, **`return` is a statement, not an expression**. You cannot
write `let x = return foo;` -- and you usually don't need to. The tail
expression of a block IS the value:

```cjcl
fn solve() -> i64 {
    let x = compute();
    x * 2               // <-- this is the return value; no `return` keyword
}
```

`return` exists for early exit from inside conditional branches, not
for the normal flow.

## Common pitfall (semicolons)

A statement ends with `;`. A tail expression does NOT end with `;` --
the missing semicolon is the signal to the parser that the expression's
value is the block's value:

```cjcl
fn double(x: i64) -> i64 {
    x * 2;     // returns Unit (type error: expected i64, found Unit)
}

fn double(x: i64) -> i64 {
    x * 2      // returns i64 -- correct
}
```
