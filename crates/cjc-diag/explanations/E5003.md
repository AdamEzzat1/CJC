# Duplicate definition

## What happened

You declared two things with the same name in the same scope. This is
the general-purpose version of `E2014` (duplicate type) -- it also
fires for two functions with the same name, two `const`s, two
imports of the same symbol, etc.

```cjcl
fn solve(x: i64) -> i64 { x }
fn solve(x: f64) -> f64 { x }    // E5003 -- name `solve` taken
```

## Why it matters

CJC-Lang doesn't support function overloading by argument type. Each
name in a scope must refer to exactly one declaration. The reason: an
overloaded name introduces an "overload resolution" step that depends
on the types of all arguments, which is fragile in the presence of
implicit conversions (which CJC-Lang doesn't have) and surprising
when the wrong overload is picked.

For values like `let`, shadowing is explicitly allowed (a new `let`
of the same name doesn't fire E5003, it just shadows). But top-level
declarations don't shadow -- they collide.

## How to fix it

**Rename one:**

```cjcl
fn solve_int(x: i64) -> i64 { x }
fn solve_float(x: f64) -> f64 { x }
```

**Or use a single function that accepts a union type:**

```cjcl
fn solve(x: Any) -> Any { ... }     // pays runtime dispatch cost
```

**Or put them in different modules:**

```cjcl
// in module integers/
fn solve(x: i64) -> i64 { x }

// in module floats/
fn solve(x: f64) -> f64 { x }
```

## Common pitfall

For traits/impls with method names, the rules are different -- each
impl block defines its own scope. E5003 doesn't fire for methods with
the same name in different impl blocks.
