# Tensor shapes don't line up for this operation

## What happened

You applied a tensor operation (add, multiply, matmul, dot, broadcast)
to tensors whose shapes can't legally combine. The diagnostic shows the
shapes involved -- read them carefully, the mismatched axis is the one
to investigate.

## Why it matters

CJC-Lang's tensors are sized at the value level (the shape is part of
the runtime tensor's state, not its compile-time type). Operations
check shapes at runtime -- element-wise ops require identical shapes;
matmul requires inner dimensions to match; broadcasts follow specific
rules.

The choice to check at runtime (rather than at compile time, as some
statically-shaped tensor languages do) is a deliberate tradeoff:

- Pro: tensors with shapes determined at load time are easy to handle
- Pro: the language stays simple -- no shape-polymorphism in the type system
- Con: shape bugs surface at runtime, not compile time

CJC-Lang's mitigation is to halt deterministically the moment a shape
mismatch is detected, with a precise message about which axes don't
line up. Silent broadcast or implicit reshape would let the bug
propagate into NaN territory.

## How to fix it

**Element-wise mismatch:**

```cjcl
let a = tensor_of_shape([3, 4], ...);
let b = tensor_of_shape([3, 5], ...);
let c = a + b;     // E8004: [3,4] vs [3,5]

// fix -- align shapes (often by slicing one or reshaping)
let c = a + b.slice([0..3, 0..4]);
```

**Matmul inner dimension:**

```cjcl
// (m, k) @ (k, n) -> (m, n)
let a = tensor_of_shape([3, 5], ...);  // m=3, k=5
let b = tensor_of_shape([4, 2], ...);  // expected k=5, got k=4
let c = matmul(a, b);    // E8004: a's inner=5, b's inner=4

// fix -- align inner dimensions
let b = tensor_of_shape([5, 2], ...);   // now k matches
```

**Broadcast:**

```cjcl
// Broadcasting rule: dimensions must either match or be 1
let a = tensor_of_shape([3, 4]);
let b = tensor_of_shape([2, 4]);
let c = a + b;   // E8004: 3 vs 2 -- neither is 1

// fix -- match the shapes, or use a 1-length dim for broadcast
let b = tensor_of_shape([1, 4]);   // broadcasts to [3, 4]
```

## Common pitfall

**Transpose and matmul order matters.** `A.matmul(B.transpose())` and
`A.transpose().matmul(B)` produce different shapes (and different
mathematical results). Read the matmul signature carefully -- the
diagnostic shows the inner dimensions, which is usually enough to spot
whether one side needs transposing.

**Row-major shape interpretation.** The shape returned by `.shape()`
is `[dim_0, dim_1, ...]` in row-major order. For a 2D tensor, that's
`[rows, cols]`. If your mental model is `[cols, rows]` (column-major,
as in Fortran or MATLAB), every shape interaction will feel reversed.
