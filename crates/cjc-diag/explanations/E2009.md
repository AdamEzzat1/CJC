# A match doesn't cover every possible case

## What happened

You wrote a `match` expression whose arms don't cover all possible
values of the scrutinee. The compiler tried to prove exhaustiveness
(every input has a matching arm) and found at least one input that
would fall through to nothing.

```cjcl
match x {        // E2009 if x can be any i64
    0 => "zero",
    1 => "one",
}    // what about 2, 3, -1, ...?
```

## Why it matters

`match` is an expression -- it must produce a value for every input.
A non-exhaustive match has inputs for which no arm runs, which would
mean "produce no value." CJC-Lang doesn't accept that.

Exhaustiveness checking is also a powerful correctness tool: when you
add a new variant to an enum, every match on that enum becomes
non-exhaustive until you handle the new case. The compiler tells you
exactly which matches need updating -- you don't have to remember to
search the codebase.

## How to fix it

**Add the missing cases explicitly:**

```cjcl
match weekday {
    "Mon" => 1,
    "Tue" => 2,
    "Wed" => 3,
    "Thu" => 4,
    "Fri" => 5,
    // E2009 -- missing Sat, Sun, and "anything else"
}

// fix
match weekday {
    "Mon" => 1,
    "Tue" => 2,
    "Wed" => 3,
    "Thu" => 4,
    "Fri" => 5,
    "Sat" => 6,
    "Sun" => 7,
    _     => 0,    // catch-all for invalid input
}
```

**Add a wildcard `_` arm to cover the rest:**

```cjcl
match x {
    0 => "zero",
    1 => "one",
    _ => "other",
}
```

## Common pitfall

A wildcard `_` is the simplest exhaustiveness fix, but it has a
downside: when you later add a new enum variant or refine the type, a
match with `_` quietly absorbs it -- you don't get a compile-time
prompt to handle the new case. For matches on enums, prefer
enumerating every variant explicitly so the compiler reminds you when
the type grows.

For matches on open types (i64, String, etc.), the wildcard is usually
the right tool because there are infinitely many values.
