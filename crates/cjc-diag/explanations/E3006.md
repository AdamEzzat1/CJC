# Cannot move out of borrowed content

## What happened

You tried to move a value out of a place that's currently borrowed --
typically by destructuring a struct or pattern-matching a value
behind a reference. The borrow checker rejects the move because it
would leave the borrowed location partially uninitialized, which
would invalidate the borrow.

```cjcl
struct Pair { a: [i64], b: [i64] }

let p = Pair { a: [1, 2], b: [3, 4] };
let r = &p;
let owned_a = p.a;            // E3006 -- can't move out of borrowed p
print(r);
```

## Why it matters

When you take a borrow `&p`, the borrow refers to *the whole* `Pair`,
not just one field. Moving `p.a` out of `p` would leave `p` in a
partially-initialized state (`a` is gone, `b` still there), but the
borrow `r` still claims to refer to a complete `Pair`. The model
forbids this to keep references coherent.

The constraint is the same as `E3001` (move of borrowed value)
applied to sub-parts: you can't surgically extract one field while
the whole is borrowed.

## How to fix it

**Drop the borrow first:**

```cjcl
let p = Pair { a: [1, 2], b: [3, 4] };
{
    let r = &p;
    print(r);
}                                 // borrow ends
let owned_a = p.a;                // OK -- no live borrow
```

**Or clone the field:**

```cjcl
let p = Pair { a: [1, 2], b: [3, 4] };
let r = &p;
let owned_a = p.a.clone();        // copy, don't move
print(r);                         // still valid
```

**Or use the borrow to read instead of moving:**

```cjcl
let p = Pair { a: [1, 2], b: [3, 4] };
let r = &p;
let a_ref = &r.a;                 // borrow into a borrow
print(a_ref);
```

## Common pitfall

Pattern matching in `match` arms doesn't always move -- some patterns
bind by reference automatically when the scrutinee is a reference. If
you see `E3006` in a match arm, check whether the scrutinee is a
borrow or an owned value, and whether the pattern is using `ref` or
move semantics.
