# A variable shadows one in an outer scope

## What happened

You declared a new variable with `let` (or `let mut`) using a name
that's already bound in an enclosing scope. The new binding **shadows**
the outer one inside this block:

```cjcl
let x = 1;
{
    let x = 2;     // W0004 -- shadows the outer `x`
    print(x);       // prints 2
}
print(x);            // prints 1 -- outer scope unaffected
```

## Why it matters

Shadowing is **intentional and supported** in CJC-Lang -- it's a
useful pattern for "transform a value through a series of bindings"
without declaring everything `let mut`. But it can also be accidental:
a programmer who thinks they're updating the outer `x` is actually
creating a new local `x` that goes out of scope at the block end.

CJC-Lang warns so you notice. If the shadowing is intentional, you
ignore the warning or rename to silence it; if it's an accident,
you've caught the bug.

## How to fix it

**If the shadowing is intentional, rename or accept the warning:**

```cjcl
let x = 1;
{
    let x_doubled = 2;      // explicit different name
    print(x_doubled);
}
```

**Or use shadowing as a "rebind" idiom:**

```cjcl
let x = read_input();
let x = x.trim();           // shadowing the same name on purpose
let x: i64 = x.parse();      // each step refines the value
```

This is idiomatic CJC-Lang -- it's how you avoid `let mut` for values
that go through a transformation pipeline. The warning is acceptable
in this pattern; some teams use `#[allow(shadow)]` to silence it.

**If the shadowing was accidental, rename the inner binding:**

```cjcl
let x = 1;
{
    let y = compute();
    print(x + y);
}
```

## Common pitfall

Function parameters that share a name with module-level bindings will
also fire W0004. This is usually fine (parameters shadowing globals is
a deliberate language design pattern that prevents accidental globals
from leaking in), but worth scanning if you weren't aware.
