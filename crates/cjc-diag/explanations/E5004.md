# Import not found

## What happened

You wrote `import some_module;` but the module couldn't be located.
CJC-Lang's module resolver searched the import paths and found no
match.

```cjcl
import nonexistent_helper;    // E5004
```

## Why it matters

The module system is the only mechanism for cross-file code reuse in
CJC-Lang. When an import fails, the resolver has tried every path it
knows; the failure is definitive, not transient.

CJC-Lang's module resolution is **deterministic** -- given the same
file tree and the same search paths, the resolver produces the same
result on every run. There's no "try the network" or "fall back to a
cached version" behaviour. If a module can't be found, the program
can't compile.

## How to fix it

**Wrong path / typo:**

```cjcl
import vizoor;    // E5004 -- typo

// fix
import vizor;
```

**Module not in the search path:**

If you're running with `cjcl run --multi-file main.cjcl`, the resolver
searches relative to `main.cjcl`'s directory. A module in a different
directory won't be found unless you adjust the path:

```cjcl
import sub.helper;    // looks for sub/helper.cjcl relative to main
```

**Module not written yet:**

The simplest possible cause -- the module file doesn't exist. Create
it (`helper.cjcl`) with at least an empty body and re-run.

## Common pitfall

If you're using `cjcl run` (single-file mode), `import` is not
supported -- the module system is multi-file only. Switch to
`cjcl run --multi-file main.cjcl`.
