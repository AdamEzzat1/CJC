# A byte-string literal was never closed

## What happened

The lexer found a byte-string opener (`b"`) but reached end-of-line or
end-of-file before finding the closing `"`. Byte-strings are CJC-Lang's
literal for raw byte sequences -- they look like strings but produce
`[u8]` arrays rather than UTF-8 `String` values.

This is the byte-string counterpart of `E0002` (unterminated regular
string literal). The cause and fix patterns are identical.

## Why it matters

Byte-strings exist as a distinct token so that the type of the literal
is visible at the source. A regular `"hello"` is a `String` (UTF-8
text); a `b"hello"` is `[u8; 5]` (raw bytes). Mixing them up at the
binding site is a common source of bugs in I/O code, so CJC-Lang
requires the prefix.

When the closer is missing, the lexer cannot tell where the byte
string was meant to end, so it errors at the opener -- the same
behaviour as `E0002`.

## How to fix it

Add the missing close quote:

```cjcl
let magic = b"GIF89a;      // E0009 -- never closed

// fix
let magic = b"GIF89a";
```

If you wanted a regular UTF-8 string and not a byte-string, drop the
`b` prefix:

```cjcl
let greeting = b"hello";   // [u8; 5], not a String
let greeting = "hello";    // String -- usually what you want
```

## Common pitfall

Byte-string escape rules are *similar* but not *identical* to string
escapes:

- `\n`, `\t`, `\r`, `\\`, `\"`, `\0` all work in both
- `\x{HH}` works in both
- `\u{HHHH}` is forbidden in byte strings (bytes can't represent
  multi-byte Unicode scalars directly)

A `\u{...}` escape inside `b"..."` will fire `E0010` (invalid byte
literal) rather than producing a multi-byte sequence.
