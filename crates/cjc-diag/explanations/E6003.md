# Conflicting trait implementations

## What happened

You declared two `impl` blocks for the same (trait, type) pair, or
two impls that overlap in the types they cover. The compiler can't
choose which one to use.

```cjcl
impl Display for i64 { ... }
impl Display for i64 { ... }    // E6003 -- duplicate impl for (Display, i64)
```

## Why it matters

CJC-Lang requires impls to be **coherent**: at most one impl exists
for any given (trait, type) pair. This is what guarantees the compiler
can resolve `value.method()` calls deterministically -- there's
exactly one method to call.

Languages without coherence (some early Java generics, some C++
templates) end up with rules like "most specific match wins" or
"first declared wins" that lead to surprising and order-dependent
resolution. CJC-Lang refuses the conflict instead.

## How to fix it

**Merge the impls into one:**

```cjcl
impl Display for i64 {
    fn fmt(&self) -> String {
        // single, canonical implementation
        i64_to_string(*self)
    }
}
```

**Or delete the duplicate** if one is a leftover from refactoring.

**Or restrict one impl** if they cover different cases via different
generic constraints:

```cjcl
impl<T: Display> Display for Vec<T> { ... }
impl<T: Debug + Display> Display for Vec<T> { ... }   // overlaps -- E6003

// fix -- pick the more general one and delete the overlapping one
impl<T: Display> Display for Vec<T> { ... }
```

## Common pitfall

Cross-crate (cross-module) coherence: even if your impls are in
different files, they conflict if they cover the same (trait, type).
The compiler treats the entire program as one coherence universe.
