# Stack overflow

## What happened

Your program's call stack grew beyond the runtime's limit and was
halted. This almost always means unbounded recursion: a function calls
itself (directly or transitively) without a base case that returns.

```cjcl
fn factorial(n: i64) -> i64 {
    n * factorial(n - 1)   // missing base case -- recurses forever
}
```

## Why it matters

CJC-Lang doesn't have a giant heap-allocated growable call stack -- it
uses the OS thread's stack, which has a hard limit (typically 8MB on
Linux/Mac, 1MB on Windows). Unbounded recursion blows past that
quickly, and the runtime halts deterministically rather than letting
the OS produce a SIGSEGV.

The reason halting is preferable to "just make the stack bigger":
unbounded recursion is virtually always a bug, and detecting it
quickly is more useful than letting it consume more memory before
crashing.

For deterministic computation, stack depth is part of the program's
resource use -- a program that overflows the stack on one machine
should overflow predictably on another, which is what this error
guarantees.

## How to fix it

**Add a base case to recursive functions:**

```cjcl
fn factorial(n: i64) -> i64 {
    if n <= 1 {
        return 1;            // base case
    }
    n * factorial(n - 1)
}
```

**Use iteration instead of recursion for deep cases:**

```cjcl
fn factorial(n: i64) -> i64 {
    let mut acc = 1;
    let mut i = 2;
    while i <= n {
        acc = acc * i;
        i = i + 1;
    }
    acc
}
```

**For genuinely deep recursion (tree traversal, etc.), consider tail
calls** -- CJC-Lang has a tail-call optimization that converts
self-recursive tail calls into loops, avoiding stack growth. The MIR
executor specifically supports this (see `MirExecError::TailCall`).

For tail-call to fire, the recursive call must be the LAST thing the
function does -- no work after it:

```cjcl
// tail-call form -- doesn't grow the stack
fn sum_to(n: i64, acc: i64) -> i64 {
    if n == 0 { acc } else { sum_to(n - 1, acc + n) }
}
```

## Common pitfall

Mutually recursive functions (`a` calls `b` calls `a`) also overflow.
The tail-call optimization currently only handles direct self-calls;
mutual recursion still grows the stack.
