# Invalid numeric literal

## What happened

The lexer found something that looked like the start of a number (a
digit, or `0x`/`0b`/`0o` prefix) but couldn't parse the rest as a
valid literal. The diagnostic points at the offending characters.

This fires for cases like:
- `12abc34` -- letters embedded in what should be digits
- `1.2.3` -- two decimal points
- `1e` -- exponent marker with no exponent
- `0x` -- hex prefix with no hex digits (also see `E0007`)

## Why it matters

CJC-Lang's numeric literal grammar is fixed and unambiguous. There's
no implicit "try parsing as something else" behaviour -- a malformed
number is rejected at the lexer stage so it never becomes a confusing
expression downstream.

The strict grammar also keeps numerical semantics predictable. A
literal like `0.1` is exactly the f64 representation of one-tenth (the
closest representable value); the lexer never re-rounds or reinterprets.

## How to fix it

**Stray characters:**

```cjcl
let x = 12abc;     // E0003 -- `abc` cannot be part of a number

// fix
let x = 12;
```

**Two decimal points:**

```cjcl
let pi = 3.14.15;  // E0003 -- which one is the decimal?

// fix
let pi = 3.1415;
```

**Incomplete exponent:**

```cjcl
let big = 1e;      // E0003 -- exponent value missing

// fix
let big = 1e6;
```

**Float with no fractional digits:**

```cjcl
let half = 5.;     // E0003 -- need at least one digit after `.`

// fix
let half = 5.0;
```

## Common pitfall

If you wrote `1_000_000` (underscores as digit separators), that **is**
valid -- the lexer treats `_` as ignored in digit position. But
`1_000_` (trailing underscore) or `_1000` (leading underscore) usually
isn't. Put underscores between digits, not at the boundaries.
