# Function parameters need explicit types

## What happened

You wrote a function whose parameter has no type annotation, like this:

```cjcl
fn solve(x) {
    x * 2
}
```

CJC-Lang stopped at `x` because it couldn't tell what kind of value `x` is
supposed to be — an integer, a float, a tensor, something else.

## Why it matters

CJC-Lang treats parameter types as part of the **determinism contract**.
Two reasons this is enforced, not just suggested:

1. **Different types reduce differently.** A sum of `f32` values and a sum
   of `f64` values give different bit patterns even for the same logical
   numbers, because the rounding boundaries are different. If a function's
   parameter type can drift between calls, its output drifts with it —
   silently breaking the "same seed, same output" guarantee.

2. **Different types pick different kernels.** When the compiler sees
   `x: f64`, it routes through Kahan-compensated summation. For `i64` it
   uses overflow-checked integer paths. Without the annotation it can't
   pick, so the function can't be lowered at all.

The annotation isn't paperwork — it's the contract that lets the rest of
the toolchain prove the function behaves the same way every time.

## How to fix it

Add the type after the parameter name:

```cjcl
// before — E1003 on `x`
fn solve(x) { x * 2 }

// after — concrete numeric type
fn solve(x: i64) -> i64 { x * 2 }

// or, if the function genuinely should accept anything
fn solve(x: Any) -> Any { ... }
```

Notice that the return type after `->` is also explicit. This is intentional:
you read the function's contract at the signature line, not by tracing the
body.

If you're using `Any`, prefer not to. `Any` is an escape hatch for genuinely
polymorphic code (e.g., generic data-structure helpers). For numerical code,
pick a concrete type — the perf and determinism story is stronger.

## Common pitfall

If you're coming from Python, TypeScript, or other languages where parameter
types are optional or inferred at the call site, the requirement here can
feel heavy. The mental shift: in CJC-Lang, the function's type signature is
its *promise* to every caller. A function with no parameter types has no
promise to make, and that's not a useful function in a deterministic system.

If you're coming from Rust, note that CJC-Lang doesn't infer parameter
types from usage — unlike closures, top-level `fn` declarations always
require annotations. This matches Rust's rule for `fn`, not its rule for
`|x| x + 1`.
