# Unused type parameter

## What happened

You declared a generic type parameter on a function, type, or impl,
but never used it. The compiler can't infer or instantiate an unused
parameter, so it flags it as an error.

```cjcl
fn solve<T>(x: i64) -> i64 { x * 2 }    // E6004 -- T is declared but never used
```

## Why it matters

Generic type parameters are part of a function/type's signature: each
parameter `T` represents a slot the caller fills with a concrete
type. An unused parameter is at best confusing (callers wonder why
they have to specify it) and at worst hides a bug -- you may have
intended to use it somewhere and forgotten.

CJC-Lang treats this as an error, not a warning, because the
function's signature with `T` and the function's signature without
`T` are different APIs. You should commit to one.

## How to fix it

**Remove the unused parameter:**

```cjcl
fn solve(x: i64) -> i64 { x * 2 }    // no generics
```

**Or use the parameter** somewhere in the function:

```cjcl
fn solve<T: Default>(x: i64) -> T {
    T::default()                       // T is used in the return type
}
```

**Or use a phantom-data marker** if you genuinely need the parameter
for type-level reasoning without runtime use:

```cjcl
fn solve<T>(x: i64, _marker: PhantomData<T>) -> i64 { x * 2 }
```

Phantom data tells the type system "this function is parameterized
over T even though no value of type T appears at runtime" -- useful
for newtype wrappers and ownership marking.

## Common pitfall

If you declared `<T>` because you were planning to use it later and
haven't gotten there yet, E6004 is a useful reminder. Add the
parameter when the code that uses it lands, not before.
