# Divided (or modulo'd) by zero

## What happened

A runtime division or modulo operation was attempted with a divisor of
zero. CJC-Lang detected this and halted the program rather than
producing a non-finite result (NaN or +/-Inf for floats; trap for
integers).

## Why it matters

Different languages handle divide-by-zero differently:

- C/C++ (integers): undefined behaviour, often a hardware trap
- Java/Rust (integers): runtime panic
- IEEE 754 floats (everywhere): produces +/-Inf or NaN silently

CJC-Lang's choice is to **always halt deterministically** on a zero
divisor -- for both integers and floats. The reason: a silent +/-Inf or
NaN in a numerical computation cascades. A single zero in one
intermediate often produces NaN in every downstream calculation, and
finding which division was the cause requires bisecting hours later.
Halting at the point of division surfaces the problem immediately.

This is part of CJC-Lang's broader stance: silent numerical failures
cost more debugging time than they save. If you *want* IEEE +/-Inf
behaviour, you can opt in with explicit guard code; the default is
fail-fast.

## How to fix it

Guard the division with a check:

```cjcl
fn safe_mean(values: [f64]) -> f64 {
    if array_len(values) == 0 {
        return 0.0;
    }
    sum(values) / (array_len(values) as f64)
}
```

For cases where zero is a legitimate signal:

```cjcl
fn rate(events: i64, seconds: i64) -> f64 {
    if seconds == 0 {
        return 0.0;   // explicit: zero seconds means zero rate
    }
    (events as f64) / (seconds as f64)
}
```

If you genuinely *do* want IEEE division semantics (+/-Inf for non-zero
/ zero, NaN for 0 / 0), do the special-casing explicitly so the intent
is visible at the source line:

```cjcl
fn ieee_div(a: f64, b: f64) -> f64 {
    if b == 0.0 {
        if a == 0.0 { 0.0 / 0.0 }       // explicit NaN
        else        { 1.0 / 0.0 * a.signum() }   // explicit +/-Inf
    } else {
        a / b
    }
}
```

In most cases the right answer is to fix the upstream reason the
divisor became zero, not to silence the error.

## Common pitfall

**Floating-point comparison to zero is itself tricky.** `if x != 0.0`
is true for `x = 1e-300` (subnormal but nonzero), and dividing by
`1e-300` produces a huge but finite number, not +/-Inf. If your guard
is meant to protect against "effectively zero," use a tolerance:

```cjcl
if abs(divisor) < 1e-10 {
    return 0.0;
}
```

## Common pitfall (modulo)

`a % b` halts on `b == 0` for the same reason as division. The fix is
the same -- guard the divisor.
