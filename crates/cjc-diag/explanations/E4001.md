# A GC-using operation appeared in a NoGC-verified path

## What happened

Inside a function marked `#[nogc]` (or reachable from one), the compiler
detected an operation that would allocate from the garbage-collected heap.
The diagnostic underlines the offending expression — most commonly a fresh
array literal, a `Vec::push`, a `format!` call, or anything that conjures a
new owned value out of thin air.

## Why it matters

The NoGC verifier is one of CJC-Lang's strongest static guarantees. Code
marked `#[nogc]` is proven, at compile time, never to:

- trigger an allocation pause,
- walk the heap in a tracing scan,
- vary in timing between runs because of heap fragmentation.

These properties matter most in the *innermost* loops of numerical code —
matmul kernels, autodiff backward passes, ABNG per-row training, the
inside of a PINN epoch. If even one allocating operation slips through,
the timing guarantee evaporates: a single GC pause inside a tight loop
can move latency from microseconds to milliseconds, and worse, do it
non-deterministically.

The verifier is **conservative** — it rejects anything it can't prove
safe, even if a clever runtime trick would have made it work in practice.
That's the point. NoGC is a promise to callers, and the compiler is the
party that keeps the promise honest.

## How to fix it

Almost every NoGC fix follows the same pattern: **pass the buffer in
instead of allocating it inside**. NoGC code is "give me a buffer, fill
it, return it" — closer to BLAS than to functional style.

```cjcl
// before — allocates a fresh Vec, E4001 on the literal
fn solve(x: Tensor) -> Tensor /nogc {
    let temp = [1.0, 2.0, 3.0];
    x + temp
}

// after — caller supplies the scratch buffer
fn solve(x: Tensor, scratch: Tensor) -> Tensor /nogc {
    scratch.copy_from(x);
    // ...mutate scratch in place...
    scratch
}
```

For string-like cases (debug output inside a hot path), the convention is
to emit a pre-formatted static string rather than building one at runtime:

```cjcl
// before — format! allocates
fn check(x: f64) /nogc {
    if x < 0.0 { panic(format!("bad x: {}", x)); }   // E4001
}

// after — fixed message, branch decides whether to fire it
fn check(x: f64) /nogc {
    if x < 0.0 { panic("solve.check: x must be non-negative"); }
}
```

If you genuinely *need* allocation in the path you're working in, the
right answer is to **remove the `#[nogc]` annotation** from this function
and any caller that depends on its NoGC-ness. Don't fight the verifier
to keep an annotation that no longer holds.

## Common pitfall

The biggest surprise for newcomers is that **collection-returning builtins
allocate**. `array_push(arr, val)` returns a new array (this is the CJC
convention — `arr = array_push(arr, val)`); inside `#[nogc]`, that
returns-a-new-array is an allocation.

The NoGC equivalents work in place:

| Allocating (forbidden in NoGC) | In-place (NoGC-safe) |
|---|---|
| `array_push(arr, v)` | mutate a pre-sized buffer + length counter |
| `format!(...)` | pre-formatted static strings |
| array/tensor literal `[...]` | pre-allocated scratch, fill it |
| `string + string` | pre-allocated string buffer |

If your code keeps wanting to allocate, that's a signal the NoGC boundary
is in the wrong place. Move the annotation outward to the function that
*calls* the hot loop, not the loop itself, and let the inner code allocate
freely if it must.
