# SSA verification failed

## What happened

After MIR lowering, the compiler runs a verifier that checks the IR is
in **single static assignment** form -- every register is assigned
exactly once across the function. The verifier found a register that's
either:

- Assigned more than once (multi-assignment violation)
- Used without being assigned first (use-before-def violation)
- Used in a different scope from its definition (dominance violation)

The diagnostic identifies the offending register and the kind of
violation.

This is an **internal compiler error** -- it indicates that MIR
construction produced ill-formed IR, not that your source code is
wrong.

## Why it matters

SSA is the bedrock that the MIR optimizer and the register-machine
executor both depend on. The optimizer uses SSA-style def-use
analysis to drive constant folding, dead code elimination, and CFG
simplification. The executor uses SSA register liveness to know when
registers can be reused. If SSA is broken, both stages produce wrong
results.

The verifier exists to catch these violations the moment they appear,
not after they've corrupted downstream output. CJC-Lang's parity
discipline (AST-eval vs MIR-exec agreement) makes SSA correctness
load-bearing: a violation here would surface as a parity-test failure
or worse, a silently-wrong result.

## How to fix it

**Report it.** E7002 is a compiler bug. File an issue at
[`https://github.com/AdamEzzat1/CJC/issues`](https://github.com/AdamEzzat1/CJC/issues)
with the failing source.

**Workaround**: restructure the source to avoid the construct that
triggers the violation. Closures, complex patterns, or nested
control flow are usually involved -- splitting the code into smaller
functions often dodges the issue.

## Common pitfall

E7002 can fire on optimizer output too (post-CF, post-DCE), not just
fresh MIR. If the optimizer is enabled (`cjcl run --mir-opt`),
disable it and see if E7002 goes away -- if it does, the bug is in
the optimizer, not in MIR construction.
