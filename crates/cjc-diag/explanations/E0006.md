# Invalid regex literal

## What happened

The lexer found a regex-literal opener (`r"`) but couldn't parse the
rest as a valid regex. Common causes:

- The closing `"` is missing
- The pattern contains a CJC-regex syntax CJC-Lang doesn't support
- An escape inside the pattern is malformed

## Why it matters

CJC-Lang has a built-in NFA-based regex engine (`cjc-regex`) and a
distinct literal syntax `r"pattern"` for it -- separate from regular
string literals. This separation matters because:

- Regex patterns have their own escape rules (`\d`, `\w`, `\(`, etc.)
  that differ from string escapes (`\n`, `\t`).
- The regex literal is validated at compile time -- a malformed pattern
  fails immediately rather than at first use at runtime.

The strict literal form also keeps regex patterns visible at the
syntactic level: a tool reading the source can find every regex in the
program by looking for `r"..."`.

## How to fix it

**Missing close quote:**

```cjcl
let p = r"\d+;      // E0006 -- regex never closed

// fix
let p = r"\d+";
```

**Unsupported pattern feature:**

CJC-Lang's regex engine supports a *subset* of standard regex: literals,
character classes, anchors, quantifiers, groups, alternation. It does
NOT support lookahead/lookbehind, named captures, or recursive patterns.
Check `crates/cjc-regex/src/lib.rs` for the exact supported syntax.

```cjcl
let p = r"(?<=foo)bar";   // E0006 -- lookbehind not supported

// fix -- restructure to avoid the assertion
let p = r"foo(bar)";       // capture group instead
```

## Common pitfall

Regex backslashes are NOT escape sequences in the lexer sense. Inside
`r"..."`, `\n` is the regex pattern "match a newline character," not
the string escape "literal newline." This is the whole reason regex
literals are syntactically distinct from strings.

If you need a literal backslash *in the regex pattern itself*, write
`\\` -- the regex engine interprets that as a single backslash to
match.
