# Invalid character inside a numeric literal

## What happened

The lexer was scanning a numeric literal when it hit a character that
isn't a valid digit for the literal's base, isn't an underscore
(digit separator), and isn't a recognized type suffix.

```cjcl
let x = 0xFG;       // E0008 -- `G` isn't a hex digit
let y = 0b201;      // E0008 -- `2` isn't a binary digit
let z = 12g34;      // E0008 -- `g` isn't anything mid-number
```

## Why it matters

Each numeric base has a fixed set of valid digits:

- Decimal: `0-9`
- Hex (`0x`): `0-9`, `a-f`, `A-F`
- Binary (`0b`): `0-1`
- Octal (`0o`): `0-7`

CJC-Lang validates this strictly because off-base digits are almost
always typos. Letting `0b9` quietly become `0` or `9` would hide bugs;
rejecting it surfaces the typo immediately.

This is distinct from `E0003` (invalid numeric literal) in that E0008
fires when the literal *is* recognizable as a numeric attempt -- the
prefix and at least some digits parse correctly -- but a foreign
character interrupts the digit sequence.

## How to fix it

Use the right digit for the base, or change the base to one that
accepts the digit:

```cjcl
let x = 0xFF;       // hex digits go up to F
let y = 0b111;      // binary digits are 0 or 1 only
let z = 1234;       // decimal needs no prefix
```

**Type suffix vs digit confusion.** CJC-Lang doesn't currently use
inline type suffixes (like `42i64` in Rust). If you wrote that, the
`i` is read as a digit position, not a suffix, so E0008 fires. Use
explicit type annotations on the binding instead:

```cjcl
let x = 42i64;      // E0008 -- no inline suffix

// fix
let x: i64 = 42;
```

## Common pitfall

When migrating from a language with hex-suffix conventions like `42h`
or `0FFh`, expect E0008. CJC-Lang uses prefixes (`0x`), not suffixes.
