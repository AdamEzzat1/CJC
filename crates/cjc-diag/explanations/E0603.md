# Snapshot layout hash mismatch

## What happened

The layout hash in the snapshot doesn't match the byte-layout the
current program expects for the stored value. This is more granular
than `E0602` (type hash): two type definitions can have the same
*shape* but different *layouts* (different padding, different field
ordering at the byte level, different endianness).

## Why it matters

The layout hash captures everything about how a value is laid out in
memory: field offsets, padding bytes, alignment requirements, integer
endianness. CJC-Lang's snapshots are byte-for-byte exact, so even
small layout shifts make a snapshot uninterpretable.

For a deterministic language, layout stability is a guarantee CJC-Lang
makes about its own data structures. The layout hash is the
verification token -- it lets the loader confirm "this snapshot was
written by a program with my exact layout assumptions."

## How to fix it

**Most layout drift comes from compiler/platform changes**, not from
source-level changes. Common causes:

- A new compiler version changed default alignment
- The snapshot was saved on a 64-bit platform and loaded on 32-bit
  (or vice versa)
- Struct field reordering -- changes layout even if the type hash
  agrees on shape

For the platform case, the answer is to use the same platform on save
and load. For the alignment case, CJC-Lang's design tries to be
explicit about layouts (`#[repr(C)]` or equivalent), but compiler
upgrades can still nudge things.

**The right answer in all cases is to re-save the snapshot** in the
current environment. Cross-platform snapshot interop is hard; the
hash mismatch correctly tells you it's not safe.

## Common pitfall

If you see `E0603` but `E0601` and `E0602` are fine, the source code
hasn't changed but the layout has. Almost always a platform or
compiler difference. Check `cjcl --version` between the save and
load environments.
