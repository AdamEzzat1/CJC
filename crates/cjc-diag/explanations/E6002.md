# Missing trait implementation

## What happened

You called a method or used an operator that requires a trait
implementation, but the type doesn't have one. This is closely
related to `E6001` (trait bound not satisfied) -- `E6002` specifically
fires when the missing impl is for a method/operator, while `E6001`
fires for declared generic bounds.

```cjcl
struct Score { value: f64 }

let a = Score { value: 1.0 };
let b = Score { value: 2.0 };
let c = a + b;     // E6002 -- Score doesn't implement Add
```

## Why it matters

Operators (`+`, `-`, `==`, etc.) in CJC-Lang are syntactic sugar for
trait methods (`Add::add`, `Sub::sub`, `Eq::eq`). For built-in types,
the impls ship with the language; for user-defined types, you have to
provide the impl explicitly.

This is a design choice that goes both ways: types don't get implicit
operator behaviour they didn't ask for (preventing surprise
overloading), but you have to opt in every time. The "opt in"
phrasing is the right framing -- it's not a tax, it's a deliberate
extension.

## How to fix it

**Implement the trait:**

```cjcl
impl Add for Score {
    type Output = Score;
    fn add(self, other: Score) -> Score {
        Score { value: self.value + other.value }
    }
}

// now `a + b` works
let c = a + b;
```

**Or use a different operation that doesn't require the missing impl:**

```cjcl
let c = Score { value: a.value + b.value };   // build directly
```

## Common pitfall

If you're adding `+` for your own type, decide whether `Add` should
also be derivable from arithmetic equality (`PartialEq`, `Ord`).
Often related traits go together; implementing one without the others
leads to surprising errors when callers expect the suite.
