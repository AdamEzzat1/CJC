# CFG verification failed

## What happened

The compiler's control-flow-graph verifier found a problem with the
MIR's basic-block structure. Common violations:

- A block has an unreachable predecessor / successor entry
- A `goto`/`branch` instruction points at a non-existent block
- A block has multiple terminators (two `return`s in a row, etc.)
- The CFG entry block has predecessors (it shouldn't -- entry is unique)

The diagnostic identifies the offending block and the rule violated.

Like `E7001` and `E7002`, this is an **internal compiler error** --
the source code is fine; the lowering produced ill-formed IR.

## Why it matters

The control-flow graph is the structural skeleton of every function in
MIR. The optimizer uses it to identify dead blocks (for DCE), to
constant-fold across branches, to compute dominance for SSA. The
executor uses it to drive next-block selection.

If the CFG is malformed, every downstream analysis silently goes
wrong. The verifier catches the malformation at MIR-construction time
rather than letting it propagate.

## How to fix it

**Report it.** E7003 is a compiler bug. File at
[`https://github.com/AdamEzzat1/CJC/issues`](https://github.com/AdamEzzat1/CJC/issues)
with the source and the version.

**Workaround**: avoid the construct that triggers the failure.
Complex `match` patterns, deeply-nested control flow, or unusual
break/continue combinations are usually involved.

## Common pitfall

CFG verification can fail after optimizer passes too, not just at
construction. If `cjcl run --mir-opt` produces E7003 but `cjcl run`
(no optimizer) doesn't, the optimizer broke the CFG.

The verifier's specific message ("orphan block," "missing successor,"
etc.) is the most useful clue -- include it in your bug report.
