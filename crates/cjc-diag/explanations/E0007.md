# A numeric prefix has no digits after it

## What happened

You wrote a numeric prefix -- `0x` (hex), `0b` (binary), or `0o`
(octal) -- but didn't follow it with any digits. The prefix tells the
lexer what base to interpret the digits in; without any digits, there's
no number.

```cjcl
let h = 0x;     // E0007 -- hex prefix with no digits
let b = 0b;     // E0007
let o = 0o;     // E0007
```

## Why it matters

The prefix is part of the lexical form of the number, not a separate
token. CJC-Lang doesn't have a way to write "the prefix `0x` as a
two-character string" mid-expression, so a bare prefix is unambiguously
a malformed numeric literal -- not, say, the start of an identifier.
The lexer rejects it explicitly rather than letting it cascade into
weird errors downstream.

This is a close cousin of `E0003` (invalid numeric literal) -- E0007 is
the specific case where the prefix is the only thing present.

## How to fix it

Add the digits:

```cjcl
let h = 0xFF;       // 255
let b = 0b1010;     // 10
let o = 0o17;       // 15
```

If the prefix appeared by accident (e.g., a stray `0x` from a partial
paste), delete it:

```cjcl
let count = 0x;     // E0007 -- delete the `0x`

// fix
let count = 0;
```

## Common pitfall

**Mixed-base typos.** Writing `0X` (capital X) instead of `0x` may
work in some languages but not in CJC-Lang -- the prefix is
case-sensitive lowercase. Same for `0B` and `0O`. Use lowercase.

**Underscores in the prefix.** `0x_FF` is generally NOT valid -- the
underscore separator must be *between* digits, not between the prefix
and the first digit. Write `0xFF` or `0xFF_FFFF`.
