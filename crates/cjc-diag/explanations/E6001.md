# Trait bound not satisfied

## What happened

You used a type in a position that requires it to implement a
particular trait, but that type doesn't satisfy the required bound.
The diagnostic shows the trait, the type, and where the bound came
from.

```cjcl
fn print_all<T: Display>(items: [T]) { ... }

struct Opaque { x: i64 }

print_all([Opaque { x: 1 }]);    // E6001 -- Opaque doesn't impl Display
```

## Why it matters

Generics in CJC-Lang are **bounded**: every type parameter declares
the operations it requires, and callers can only pass types that
satisfy those bounds. This is monomorphization-friendly (the compiler
generates a specialized version per concrete type) and produces
errors at the call site rather than deep inside the generic code.

Bounded generics are also what makes CJC-Lang's deterministic-by-
default story compose with abstraction: the compiler knows exactly
which trait methods will be called, so it can prove the same
properties (Kahan summation, no FMA, etc.) hold for the specialized
function.

Note: generics may not be fully implemented in your CJC-Lang version
yet. Codes E6xxx exist as placeholders for that future work.

## How to fix it

**Implement the required trait for your type:**

```cjcl
impl Display for Opaque {
    fn fmt(&self) -> String {
        f"Opaque(x={self.x})"
    }
}

// now print_all works
```

**Or use a type that already satisfies the bound:**

```cjcl
print_all(["hello", "world"]);    // String implements Display
```

**Or remove the bound** if your function doesn't actually need it:

```cjcl
fn print_all<T>(items: [T]) { ... }    // no Display bound, can't print
```

## Common pitfall

Trait bounds compose: a function with `T: Display + Clone` requires
*both* traits. Implementing one but not the other still fires E6001.
