# Use after move

## What happened

You used a value after its ownership was moved elsewhere (passed to a
function, assigned to another variable, returned). Once a value is
moved, its original binding no longer holds a valid value -- using it
is a compile error.

```cjcl
let v = make_vec();
let w = v;           // moved v into w
print(v);            // E3002 -- v no longer holds a value
```

## Why it matters

The ownership model treats values as unique resources. Each value has
exactly one owner. When you "move" a value (assign it, pass it by
value, return it), the previous owner gives up the value. There's no
way for two bindings to hold the same value at once -- without
explicit cloning.

This rule prevents double-free bugs, use-after-free bugs, and
data-race bugs at compile time. CJC-Lang inherits this from Rust's
ownership model; the determinism guarantees benefit because every
value's lifetime is statically traceable.

## How to fix it

**Clone instead of moving** if you need both bindings:

```cjcl
let v = make_vec();
let w = v.clone();   // copy
print(v);            // still valid
print(w);
```

**Borrow if you only need to read the value:**

```cjcl
let v = make_vec();
let w = &v;          // borrow, don't move
print(v);            // still valid
print(w);
```

**Or restructure to avoid using v after the move:**

```cjcl
let v = make_vec();
print(v);            // use v first
let w = v;           // then move it
```

## Common pitfall

For primitive types (`i64`, `f64`, `bool`, char), assignment **copies**
instead of moves -- they implement `Copy`. So `let a = 5; let b = a;
print(a);` works without E3002. This rule only catches non-Copy types
(arrays, strings, structs).
