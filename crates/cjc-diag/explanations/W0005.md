# A deprecated feature was used

## What happened

You used a language construct, builtin function, or syntax that's
marked deprecated in the current CJC-Lang version. Deprecated features
still work -- the warning fires, the program continues -- but they're
on their way out of the language. The diagnostic usually includes
guidance about the replacement.

## Why it matters

CJC-Lang is a moving target right now (pre-1.0). Some early-version
features turned out to be poor choices: ergonomics, semantic clarity,
or composition with later features made them worse than the
alternatives. Removing them outright would break existing code, so
they're deprecated first: marked, warned about, and scheduled for
removal in a future version.

The deprecation warning is your **lead time** to migrate. The longer
you ignore it, the more risk that an upgrade removes the feature
entirely and breaks your code.

## How to fix it

Follow the migration guidance in the warning message. The warning
usually points at:

- The replacement form to use
- The version where the deprecated feature will be removed
- The ADR or design doc explaining the change (if there is one)

Common deprecation patterns in CJC-Lang's history:

```cjcl
// old (deprecated)
let arr = make_array(5);

// new
let arr: [i64] = [0; 5];     // array literal with default value
```

If the diagnostic doesn't include the replacement, check
[`docs/CHANGELOG.md`](../../../CHANGELOG.md) for the deprecation entry.

## Common pitfall

Deprecation warnings are easy to ignore because the code still works.
Resist the temptation. CJC-Lang's deprecation timeline is usually
short (the language is still small and the maintainer is a single
person -- "remove next version" means months, not years).
