# Trait method signature mismatch

## What happened

You wrote an `impl Trait for Type` block whose method signatures
don't match the trait's declared signatures. The diagnostic shows
the trait's signature, your impl's signature, and the specific
discrepancy.

```cjcl
trait Greet {
    fn hello(&self) -> String;
}

struct Person { name: String }

impl Greet for Person {
    fn hello(&self) -> i64 {       // E6006 -- trait says String, impl says i64
        42
    }
}
```

## Why it matters

The trait is a contract; the impl must satisfy it exactly. Callers
of `Greet::hello` rely on getting a `String` back, regardless of the
concrete type. An impl that returns `i64` instead would break every
caller that didn't know.

CJC-Lang checks the impl signature against the trait declaration to
catch this at the impl site, not at every call site. This is the same
contract-enforcement pattern that's at the heart of the whole type
system.

## How to fix it

**Match the trait's signature exactly:**

```cjcl
impl Greet for Person {
    fn hello(&self) -> String {       // String, matching the trait
        f"Hello, {self.name}!"
    }
}
```

The signature includes:
- Parameter count and types
- Parameter mutability (`&self` vs `&mut self` vs `self`)
- Return type
- Generic parameters and their bounds
- Effect annotations (if present)

All of these must match.

## Common pitfall

`&self` vs `&mut self` vs `self` are three different signatures, even
though they all "look like a method." If your impl uses one and the
trait uses another, E6006 fires. Read the trait's declared receiver
carefully.

Effect annotations on trait methods also matter -- a trait method
declared `/det` requires the impl to also be deterministic.
