# An IO operation appeared in a pure context

## What happened

Inside a function marked pure (no `/io` effect) -- or reachable from
one -- the compiler detected an operation that performs IO: reading a
file, writing to stdout, opening a network connection, getting the
current time.

```cjcl
fn solve(x: f64) -> f64 {
    print("debug: x = {x}");    // E4002 -- solve() has no /io effect
    x * 2.0
}
```

## Why it matters

Effects in CJC-Lang are part of the function's contract. A function
without `/io` promises its callers it doesn't perform side effects on
the outside world. Letting an `/io`-needing operation slip through
would break that promise silently.

This matters for several CJC-Lang properties:

- **Determinism**: IO usually depends on external state (file
  contents, network responses, the clock) that varies between runs.
  A pure function can be re-evaluated freely without changing the
  program's behaviour; an IO function cannot.
- **Audit**: in regulated contexts, "did this function touch the
  outside world?" is a question with legal weight. The effect
  annotation makes the answer visible at the signature.
- **Parallelism**: pure functions are safe to evaluate in any order
  or in parallel; IO functions need careful sequencing.

## How to fix it

**Declare the IO effect on the function:**

```cjcl
fn solve(x: f64) -> f64 /io {
    print("debug: x = {x}");
    x * 2.0
}
```

**Or remove the IO from the function:**

```cjcl
fn solve(x: f64) -> f64 {
    x * 2.0     // pure -- no IO
}

// IO happens at the call site, which is /io
fn main() /io {
    let x = 3.14;
    let y = solve(x);
    print(f"y = {y}");
}
```

The second pattern is usually preferred: keep IO at the program's
edges, leave the computational core pure.

## Common pitfall

Some operations look pure but aren't: `random()` (depends on RNG
state), `current_time()` (clock), even `print()` (stdout). Anything
that observes or modifies state outside the function's arguments is
IO. The effect annotation makes this rigorous.
