# CJC-Lang cannot infer the type of this expression

## What happened

The type checker reached an expression or binding where it couldn't
determine the type from context alone. Usually this happens at a `let`
binding where:

- The value is a polymorphic expression (e.g., `let v = []` -- empty
  arrays don't have a default element type)
- The right-hand side is `None` or another generic constructor
- The expression is part of a chain where the next step would need to
  know this type before it could itself be typed

## Why it matters

CJC-Lang uses **local type inference** -- the type of an expression is
determined by its immediate context, not by tracing the whole program.
The benefit: a single type error is local to a specific source line,
not "somewhere in your program." The cost: you sometimes need to write
a type annotation where another language would let you skip it.

The choice is deliberate. Whole-program type inference (Haskell-style)
is powerful but makes errors hard to localize -- a small change in one
function can produce inscrutable errors three files away. CJC-Lang
prefers local errors with explicit annotations.

Type annotations also feed into the determinism contract: knowing the
type at the binding site lets the compiler pick the right numerical
kernel (Kahan for `f64`, overflow-checked paths for `i64`) and bake
that choice into the MIR.

## How to fix it

Add a type annotation:

```cjcl
// cannot infer type
let v = [];          // E2002 -- empty array of what?

// fix
let v: [i64] = [];
```

```cjcl
// cannot infer type for a polymorphic call
let result = parse(input);   // if parse() returns Any

// fix -- annotate to commit to a concrete type
let result: f64 = parse(input);
```

```cjcl
// cannot infer at function-return position
fn solve(x: i64) {
    if x > 0 { 1 } else { 0 }   // E2002 -- what does this return?
}

// fix -- declare the return type
fn solve(x: i64) -> i64 {
    if x > 0 { 1 } else { 0 }
}
```

## Common pitfall

If you're chaining method calls and one returns a polymorphic value,
the parser may fire E2002 at the *next* step (which can't pick a path
until the previous step's type is known). Annotate the polymorphic
intermediate to break the chain.

## Why not just use `Any`?

You can -- `let v: Any = []` will compile -- but `Any` is a runtime
type-dispatch escape hatch. Operations on `Any` go through slow dynamic
checks, and the value won't be subject to CJC-Lang's deterministic
numerical guarantees (which depend on knowing the concrete numeric
type). Reserve `Any` for genuinely polymorphic code (generic container
helpers); for everyday code, pick a concrete type.
