# An unexpected token appeared

## What happened

CJC-Lang's parser encountered a token in your source that it doesn't know
how to interpret in this context. The diagnostic points at the specific
token; the message often hints at what was expected instead.

This is the parser's **catchall** error -- it fires when the input clearly
isn't a valid CJC-Lang program but the parser doesn't have enough context
to be more specific. Its more specific cousins are:

- `E1001` -- "expected X, found Y" (the parser knew exactly what should be there)
- `E1002` -- "expected expression" (a more focused variant with a hint)

## Why it matters

CJC-Lang has a fixed grammar. Unlike languages with whitespace-significance
(Python) or aggressive token elision (JavaScript with automatic semicolon
insertion), the parser will not silently "guess" what you meant. This is
intentional -- a parser that quietly fixes your source is a parser that can
change the meaning of your program without telling you. CJC-Lang prefers to
halt at the unfamiliar token and let you decide what was intended.

This also makes the language easier to teach: there's exactly one valid
grammar, and you can predict where parse errors will fire.

## How to fix it

The right fix depends on what's at the token's position. A few common
patterns:

**Missing semicolon between statements:**

```cjcl
let x = 1
let y = 2          // E1000 -- the `let` was unexpected here

// fix
let x = 1;
let y = 2;
```

**Stray character (often a typo or leftover from editing):**

```cjcl
fn foo() { @@@ }   // E1000 on the `@`

// fix -- remove the stray character
fn foo() { 0 }
```

**Block body where an expression was expected:**

```cjcl
let x: i64 = if cond
    1
else
    2;             // E1000 -- branches need braces

// fix
let x: i64 = if cond { 1 } else { 2 };
```

## Common pitfall

Coming from C/Java/Rust? CJC-Lang **does** use `;` to separate statements
but **does not** require `;` after block expressions inside function
bodies (`if {}`, `while {}`, `for {}`). The rule:

- Pure expressions and `let` bindings: end with `;`
- Block-statement forms (if/while/for): no trailing `;`

A common cause of E1000 is putting a `;` where none was needed -- the
parser sees the unexpected `;` as a token out of place.

## Common pitfall (closures)

CJC-Lang doesn't currently support inline anonymous function literals.
If you wrote `|x| x + 1` (a Rust closure) or `(x) => x + 1` (a JavaScript
arrow), the parser will fire E1000 on the pipe or arrow. Define a named
function instead.
