# A function was called with the wrong number of arguments

## What happened

You called a function and supplied either too many or too few arguments
compared to its declared signature. The diagnostic shows both numbers:
how many the signature expects, how many you passed.

## Why it matters

CJC-Lang doesn't have default arguments yet (it's a planned feature --
see the deferred items in `CLAUDE.md` under FEATURE IMPLEMENTATION
SCOPE). Every function declares exactly *N* required parameters, and
every call site must pass *N* arguments. The exception is **variadic**
functions declared with `...` (e.g., `fn sum(...xs: f64) -> f64`),
which accept any number including zero.

This rule keeps function call sites predictable: by reading the
function's signature you know exactly how it will be invoked, with no
implicit fills from `Option`, `nil`, or undefined.

## How to fix it

Match the call to the signature. Look at the function's declaration to
see what's required:

```cjcl
fn add(a: i64, b: i64) -> i64 { a + b }

add(1);          // E2005 -- expected 2 args, got 1
add(1, 2, 3);    // E2005 -- expected 2 args, got 3

// fix
add(1, 2);
```

If you genuinely need optional arguments today, two patterns work:

**Tuples** when the optional part has a single shape:

```cjcl
fn solve(input: (i64, i64)) -> i64 { ... }
solve((1, 2));
```

**Variadic** when the count can vary:

```cjcl
fn sum(...xs: f64) -> f64 {
    let mut total = 0.0;
    for x in xs { total = total + x; }
    total
}
sum(1.0, 2.0, 3.0);   // OK
sum();                 // also OK -- returns 0.0
```

## Common pitfall

If you migrated code from Python or JavaScript and forgot that CJC-Lang
has no default arguments, this error will fire on every call site that
*would* have used a default. The mechanical fix is to pass the value
explicitly everywhere. Default parameters are on the roadmap and will
eventually relax this requirement.

## Common pitfall (closures/higher-order)

Functions used as values keep their arity. If you pass `add` (arity 2)
where a callback of arity 1 is expected, E2005 fires at the call site
inside the higher-order function, not at the place where you passed
`add`. Look at what's calling your callback to see the expected arity.
