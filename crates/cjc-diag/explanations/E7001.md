# Internal compiler error during MIR lowering

## What happened

CJC-Lang's compiler hit a bug while lowering your code from HIR (the
high-level IR) to MIR (the mid-level IR used by the optimizer and
the register-machine executor). The diagnostic message usually
includes a description of what went wrong inside the compiler.

This is **not** an error in your code -- E7001 means the compiler
itself failed to compile a program that should be compilable. The
preceding stages (lex, parse, type-check) accepted your code, but
the lowering pass couldn't translate it.

## Why it matters

Internal compiler errors are the most embarrassing kind of failure
for a compiler. They signal that the language's invariants (the
ones the type checker relied on, or the ones MIR construction
assumed) weren't actually maintained by the upstream stages.

For CJC-Lang specifically, E7001 is concerning because the parity
discipline (AST eval vs MIR exec produce identical results) relies
on MIR construction being a total function from valid HIR. An E7001
means there's a HIR shape the lowering doesn't handle.

## How to fix it

**Report the error.** This is genuinely a compiler bug. File an issue
with:

- The CJC-Lang version (`cjcl --version`)
- A minimal reproduction (`cjcl run minimal.cjcl`)
- The full error message and any backtrace

Pointer: the report goes to
[`https://github.com/AdamEzzat1/CJC/issues`](https://github.com/AdamEzzat1/CJC/issues).

**Workaround**: while waiting for a fix, try restructuring the code to
avoid the construct that triggers the ICE. The error message often
hints at which language feature is involved (closures, generics,
patterns, etc.) -- avoiding that feature locally usually compiles.

**Don't** try to patch around an ICE by adding type ascriptions or
explicit `as` casts -- that only masks symptoms.

## Common pitfall

E7001 doesn't usually fire for plain programs; it fires for unusual
combinations of features (deep closures + match patterns + generic
calls, etc.). If you're writing very straight-line code, an E7001 is
extremely unlikely; if you're combining advanced features, you may be
exploring corners of the compiler that haven't been heavily tested.
