# Expected a type expression here

## What happened

The parser was at a position where a *type* should appear -- after `:`
in a type annotation, after `->` in a function return type, inside `<...>`
of a generic argument -- but found something that doesn't form a type.

```cjcl
fn solve(x: 42) -> i64 { x }   // E1009 -- `42` isn't a type
let v: + = 10;                  // E1009 -- `+` isn't a type
```

## Why it matters

CJC-Lang distinguishes the *type* grammar from the *expression* grammar
strictly. You can't use a value where a type is expected, and vice
versa. This is what makes type checking decidable: the compiler always
knows whether it's looking at a type or a value.

Languages that blur this line (TypeScript, Python with generic
annotations) often have surprising edge cases where the parser gets
confused. CJC-Lang's strict separation pushes the surprise into a
clean compile-time error.

## How to fix it

Use a real type name in the type position:

```cjcl
fn solve(x: i64) -> i64 { x }     // i64, f64, bool, Tensor, etc.

let v: f64 = 10.0;
```

For generic / polymorphic code, use `Any`:

```cjcl
fn maybe_print(value: Any) {
    // ...
}
```

## Common pitfall

If you migrated from a language where type names and value names share
a namespace, expect E1009 whenever you accidentally type a value name
in a type position. For example:

```cjcl
const MAX: i64 = 100;
let v: MAX = 50;     // E1009 -- MAX is a value, not a type

// fix
let v: i64 = 50;
```
