# Invalid escape sequence in a string

## What happened

Inside a `"..."` string literal, the lexer found a backslash followed
by a character that isn't one of CJC-Lang's recognized escape
sequences. Recognized escapes are:

- `\n` -- newline
- `\t` -- tab
- `\r` -- carriage return
- `\\` -- a literal backslash
- `\"` -- a literal double-quote
- `\0` -- null byte
- `\x{HH}` -- a byte with the given two-hex-digit value
- `\u{HHHH}` -- a Unicode scalar value (one to six hex digits)

Anything else after `\` is an error.

## Why it matters

CJC-Lang is strict about escape sequences for the same reason it's
strict about everything else: ambiguity is the enemy of determinism.
A language that silently accepts `"\q"` as the two-character string
`\q` is one where typos in escape sequences become hidden bugs.
Rejecting the unknown escape catches it at compile time.

## How to fix it

**Unknown escape -- usually meant a literal backslash:**

```cjcl
let path = "C:\Users\Adam";    // E0004 -- \U is not an escape

// fix -- double the backslash to mean a literal backslash
let path = "C:\\Users\\Adam";
```

**Missing escape -- meant a literal quote:**

```cjcl
let quoted = "He said "hello"";   // E0002, possibly E0004

// fix
let quoted = "He said \"hello\"";
```

**Unicode that doesn't fit `\u{...}`:**

```cjcl
let emoji = "ὠ0";   // E0004 -- need braces

// fix
let emoji = "\u{1F600}";
```

## Common pitfall

Coming from Python? Python's `r"raw strings"` don't exist in CJC-Lang.
You must always escape backslashes in regular strings. For regex
literals specifically, CJC-Lang uses `r"pattern"` syntax (see `E0006`)
where backslashes are not interpreted -- but that's a separate token,
not a string.
