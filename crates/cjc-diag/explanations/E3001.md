# Move of a borrowed value

## What happened

You tried to move ownership of a value (consume it, pass it by value
to a function, return it) while it was still being borrowed (referenced)
by another part of the program. The borrow checker rejected the move
because moving while borrowed would invalidate the existing borrow.

```cjcl
let v = make_vec();
let r = &v;          // r borrows v
let w = v;           // E3001 -- can't move v while r still borrows it
print(r);            // would have been a use-after-move
```

## Why it matters

CJC-Lang uses ownership and borrowing (inspired by Rust) to prove
memory safety at compile time without garbage collection. The
invariant: every value has exactly one owner at a time, and any
outstanding borrows must remain valid for their entire lifetime.

Moving a value while it's borrowed would create a dangling reference
-- the borrow `r` would point at memory that no longer holds a valid
value (since `v` moved away). The borrow checker catches this
statically rather than letting it become a runtime bug.

Note: borrow checking may not be fully wired up in your CJC-Lang
version yet. If you see this error, the feature is online; if you
expected to see it and didn't, the analyser may still be in
development.

## How to fix it

**Drop the borrow before moving:**

```cjcl
let v = make_vec();
{
    let r = &v;
    print(r);        // borrow ends at the end of this block
}
let w = v;           // OK -- no live borrow
```

**Or clone instead of moving:**

```cjcl
let v = make_vec();
let r = &v;
let w = v.clone();   // copy, don't move
print(r);            // still valid
print(w);
```

**Or restructure to avoid the simultaneous borrow + move:**

```cjcl
fn use_value(v: [i64]) -> i64 { v.len() }

let v = make_vec();
let n = use_value(v);   // moves v -- no longer accessible after this
// don't try to use v here
```

## Common pitfall

In CJC-Lang (like Rust), assignment moves by default for non-Copy
types. If you're used to a language where assignment is reference-
sharing (Python, JavaScript), this surprises you. The fix is usually
to call `.clone()` explicitly.
