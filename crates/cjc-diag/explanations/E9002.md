# Circular module dependency

## What happened

The module loader detected that two or more modules import each
other in a cycle. For example:

```cjcl
// module a.cjcl
import b;

// module b.cjcl
import a;
```

The diagnostic shows the cycle: `a -> b -> a` (or longer).

## Why it matters

Module load order in CJC-Lang is a partial order derived from the
import graph -- each module is fully initialized before any module
that depends on it. A cycle means there is no valid order: neither
`a` nor `b` can be initialized first.

Some languages tolerate cycles by lazy-binding references (Python,
Ruby), but this trades a compile-time error for runtime
"AttributeError" surprises depending on import order. CJC-Lang's
strict resolution catches the cycle up front, so a working program
is guaranteed to have a well-defined initialization order.

## How to fix it

**Break the cycle by extracting shared definitions.** If `a` and `b`
both depend on each other, what they share usually belongs in a
third module they both import:

```cjcl
// shared.cjcl -- contains the common definitions
struct Common { ... }

// a.cjcl
import shared;

// b.cjcl
import shared;
```

**Or remove an import** if one direction isn't really needed:

```cjcl
// a.cjcl
import b;

// b.cjcl -- WAS importing a, but didn't actually need to
// (remove the import)
```

**Or merge the two modules** if they're tightly coupled:

```cjcl
// ab.cjcl -- contains what was in a and b
```

## Common pitfall

The shortest fix is rarely the right one. A cycle usually indicates a
design issue: two modules with bidirectional knowledge are coupled
more tightly than they should be. The extract-to-shared-module fix
forces you to identify what's actually shared, which often clarifies
the design.
