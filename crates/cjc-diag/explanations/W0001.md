# A variable was declared but never used

## What happened

You wrote `let x = ...` (or a function parameter, struct field, or pattern
binding called `x`), and `x` was never read anywhere afterwards. The
compiler kept compiling — this is a **warning**, not an error — but
flagged the unused binding so you'd notice.

## Why it matters

Most unused variables are bugs:

- You computed something expensive and forgot to use the result.
- You renamed a variable in one place and the rest of the code still
  reads the old name (which is now an unused binding plus an
  unresolved-name error somewhere else).
- You stopped using a variable mid-edit and forgot to delete the line.

But sometimes the unused-ness is *intentional*: a placeholder during a
refactor, a destructured tuple field you don't care about, a parameter
required by an interface (callback signatures, trait methods). The
warning exists because either case is plausible, and the compiler can't
tell which is which.

This is one of the few warnings in CJC-Lang that's worth treating
seriously even when "it's just a warning." A persistent unused-variable
warning is almost always either a real bug or a sign that your code
hasn't quite landed yet.

## How to fix it

**If the variable is genuinely unused, delete it:**

```cjcl
let x = compute_something();   // W0001 — and the computation is wasted
let y = compute_other();
process(y);

// fix
let y = compute_other();
process(y);
```

**If the variable is intentionally unused, prefix its name with `_`.**
This is the universal CJC-Lang signal for "yes, I know this is unused,
and that's the point":

```cjcl
// silences W0001 — the leading underscore is the convention
let _placeholder = compute_something();

// or for parameters in callback signatures
fn handler(_event: Event, msg: Message) {
    print(msg);
}
```

**If you're destructuring and only want part of a tuple/struct, use `_`
for the parts you skip:**

```cjcl
let (x, _, z) = three_part_result();    // ignores the middle field
let MyStruct { name, _ } = record;      // pulls out name, ignores the rest
```

## Common pitfall

**Dead-code branches keep variables "used" in a misleading way.** If your
code looks like this:

```cjcl
let x = compute();
if false {
    use(x);
}
```

…then `x` is technically used (inside the `if false` block) and you
won't get W0001. But you'll likely get **W0003 (unreachable code)** on
the body of the `if`, which is the deeper issue. Fix W0003 first; W0001
will resolve itself or surface a real bug.

**Pattern-bound variables count.** Match arms that bind names but don't
use them produce W0001 just like `let`:

```cjcl
match opt {
    Some(x) => 1,        // W0001 on `x`
    None => 0,
}

// fix — use _ when the value doesn't matter
match opt {
    Some(_) => 1,
    None => 0,
}
```

## Note on severity

This is a warning, not an error — compilation continues, your program
runs, your tests pass. CJC-Lang doesn't currently treat warnings as fatal
by default, but tooling that audits "production-ready" code (the
forthcoming `cjcl ci` strict mode) will eventually surface persistent
warnings. Cleaning them up is cheap; deferring them adds up.
