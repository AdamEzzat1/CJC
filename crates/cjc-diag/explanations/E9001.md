# Module not found

## What happened

The module loader tried to resolve a module reference (an `import` or
a qualified name) and couldn't find a matching file or registered
module on disk.

This is the runtime / module-loader version of `E5004` (which fires at
parse-time for `import` statements). E9001 fires deeper in the module
machinery -- for transitively-referenced modules, or for module loads
that happen after parsing is done.

## Why it matters

CJC-Lang's module resolution is deterministic and file-based -- a
module name maps to a file path via the configured search rules, and
the file either exists or doesn't. There's no network fetch, no
registry lookup, no on-the-fly generation. If a module can't be
found, the program can't run.

This matters for reproducibility: a program's module dependency graph
is fully determined by the source tree at compile time. Two
invocations of the same program against the same tree always resolve
the same modules.

## How to fix it

**Verify the file exists at the expected path.** CJC-Lang resolves
`import some.module` by looking at filesystem paths relative to the
multi-file entry point.

**Check the search path** if your project layout has subdirectories:

```cjcl
// main.cjcl
import sub.helper;    // looks for sub/helper.cjcl

// sub/helper.cjcl must exist
```

**Run in multi-file mode:**

```bash
cjcl run --multi-file main.cjcl    # NOT just `cjcl run`
```

Single-file mode (`cjcl run main.cjcl`) doesn't enable the module
system.

## Common pitfall

`import` resolution happens early (E5004 is the parse-time form), but
`E9001` typically fires for module loads that aren't `import`s -- for
example, a `cjc-snap` deserialization that references a type defined
in a module not currently loaded, or a deferred module load triggered
by an indirect reference.

If you see `E9001` for a module that *is* imported, double-check that
the import is in the file the failure originates from (not just in
another file that happens to import this one).
