# CANA Cost Model Training — Findings (v1 → v2 → v3)

**Date:** 2026-06-07 (TIER B #4 from handoff + Options A/B follow-ups)
**Status:** v3 shipped — Option A (per-pass diagnostic counts) fully replaces wall-clock.
**Output:** `LinearCostModel::trained()` with v3 coefficients + 69-program corpus + permanent `apply_pass_with_diagnostics` API surface.

Three iterations of cost-model training, each one dropping RMSE on every pass:

## 1. Iteration history

### v1 (first run, baseline)
- 18-program corpus
- Wall-clock median-of-5 measurement for every pass
- All five RMSEs in 0.13-0.18 (≥1.5× the mean benefit being measured)
- Sign-flipped weights, all confidences below 0.32

### v2 ("move the needle" iteration)
- 18 → 60 programs
- Inner-loop iteration counts scaled 10-100×
- N_ITERS 5 → 21
- MIR-node-count delta proxy introduced for CF + DCE (size-shrinking passes)
- Discovered empirically that CSE and SR don't shrink MIR structure in this
  codebase — left them on wall-clock
- Result: CF improved 3.3×, DCE 1.7×, others modest

### v3 (Option A: per-pass diagnostic counts)
- New `cjc_mir::optimize::PassDiagnostics` struct + `apply_pass_with_diagnostics`
  API (the diagnostic counts are documented in the struct's doc comment).
- Each pass instrumented to return its native count of "interesting changes":
  - `strength_reduce_fn(func) -> usize` (count of `try_strength_reduce` Some)
  - `cse_fn(func) -> usize` (count of replacements applied)
  - `licm_fn(func) -> usize` (count of statements hoisted)
- Training signal becomes `changes_applied / nodes_before` — pass-native,
  deterministic, zero-variance
- 60 → 69 corpus programs (added CSE/LICM-eligible programs with
  explicit `let` bindings, after discovering the lowered MIR doesn't
  always expose duplicates the way I'd assumed)
- **Bonus**: fixed a real CSE bug while doing this (see §5)

### LICM finally has the right sign

The LICM coefficient `w_loop_depth` was **negative** in v1 and v2 (sign
flip artifact of wall-clock noise). In v3 it's **positive** (3.78e-3),
matching intuition: LICM helps more when loops are present and deeper.

This is the kind of correctness regression that survived v1 and v2
because the noise hid it. The pass-native signal exposed it.

## 3. Why the bench wall-clock dropped from minutes to seconds

v2 measurement loop:
- For each (program, wall-clock pass): run with-and-without 21 iterations each
- 60 × 3 × 2 × 21 = 7,560 program runs
- Wall-clock: ~6 minutes

v3 measurement loop:
- For each (program, pass): apply pass once, read diagnostic count
- 69 × 5 = 345 single applications
- No actual program execution required
- Wall-clock: ~10 seconds

22× wall-clock reduction with cleaner signal. The compile step still
dominates because `cargo build --release` rebuilds the binary, but the
actual measurement is now near-instantaneous.

---

## 4. Option B verdict: now unnecessary

The v1 findings doc proposed Option B (millisecond-scale workloads +
perf counters) as a fix for wall-clock noise. Option A made Option B
obsolete: **there is no wall-clock signal anywhere in v3**, so there's
no scheduler noise to reduce.

Option B would still be useful if someone wanted to **validate** the
trained coefficients against actual measured runtime on real programs.
That's an evaluation step, not a training step. Documented for future
work but not in the critical path.

## 5. Permanent compiler improvements shipped alongside v3

The Option A work created three durable changes to cjc-mir, independent of
the trained coefficient values:

### A. `PassDiagnostics` + `apply_pass_with_diagnostics` API

New public surface in `cjc_mir::optimize`. Per-pass diagnostic counts
are now first-class observability data. Future consumers:
- `cjcl emit --pass-stats` (hypothetical CLI surface)
- audit reports / `CanaReport` extensions
- regression detection (e.g. "this commit dropped CSE replacements
  from 12 to 0 on this benchmark")

Documented in the struct's doc comment with per-pass `changes_applied`
interpretation.

### B. CSE bug fix — VarLocal handling in `expr_key`

The `expr_key` function in CSE only handled `MirExprKind::Var(name)` —
the pre-slot-resolution form. After the Tier-0 slot resolution pass,
`x` becomes `MirExprKind::VarLocal { name: "x", slot: 0 }`. The
opaque-pointer fallback gave every `VarLocal` a unique key, so CSE
**silently never fired on any real lowered code**.

Same family of bug as `task_a30e8eec` (LICM helper-name variant) and
commit `2983f2b` (reduction analyzer). Fixed by adding the
`VarLocal { name, .. }` arm to `expr_key`.

Net effect on real CJC-Lang programs: CSE now actually works. Programs
with `let a = x + y; let b = x + y;` and similar patterns will see one
binding eliminated per duplicate pair.

### C. Pass-native `_fn` signatures return change counts

`strength_reduce_fn`, `cse_fn`, `licm_fn` (and their helpers) now return
the count of changes instead of `()`. This change is internal — the
public `apply_pass` wrapper preserves the original `bool` contract — but
it means any future tool that wants the count can read it without
re-instrumenting.

## 7. How to regenerate

```bash
cargo run --release -p cana-train-cost-model
```

- Phase 1 (collect): ~10 seconds (69 programs × 5 passes, no wall-clock)
- Phase 2 (OLS fit): instant
- Phase 3 (emit Rust): instant
- Total: ~10 seconds + cargo build time

**Output is bit-identical across runs.** Same machine, different machine,
same coefficients. This is the v3 reproducibility property.

To paste new coefficients: copy the printed `trained_pass_coefficients`
match arms into `crates/cjc-cana/src/linear_cost_model.rs`, replacing
the existing match arms inside that function.

---

## 8. What you can NOT trust the trained model for

It's the result of one specific corpus on one specific optimizer
implementation. It will be wrong if:

- Your real program has a much larger or differently-shaped feature
  distribution (e.g. 10,000-line functions while corpus has 200).
- A new pass is added or an existing pass is rewritten without
  re-running the training.
- The cjc-mir lowering changes the shape of generated MIR (e.g. starts
  producing `MirExprKind::Block` where it used to produce flat lets).

For all these cases the answer is: re-run the training. It takes
10 seconds.

## 10. Held-out validation (§3A.1 from the next-session handoff)

The RMSEs reported in section 5 are **training RMSE** — the model's error
on the corpus it was fit to. They tell us how well the linear cost model
*can* reproduce the signal we measured; they tell us nothing about whether
it will generalize to programs it hasn't seen.

This section runs the §3A.1 experiment: split the corpus into train (75%)
and test (25%) by program index (every 4th program → test), re-fit on
train only, evaluate on test. The gap between train-RMSE and test-RMSE,
and the ratio `test_rmse / train_rmse`, are the generalization signals.

### Methodology

- **Split rule.** `i % 4 == 0` → test set. Deterministic, stratified across
  the corpus structure (programs are grouped by pass-affinity, so the
  every-4th sampling naturally hits each affinity category roughly equally).
- **Train.** 54 programs, 595 training points.
- **Test.** 19 programs, 240 held-out points.
- **Fit.** Same `fit_ols_gd` (5000 GD steps, lr 0.05, column-max normalization).
  Train data only — the test points are never seen by the fitter.
- **Predict.** Raw-feature prediction with the train-fit coefficients
  applied to test-point features. RMSE in raw benefit units.

### Results

| Pass | N train | N test | train_rmse | test_rmse | gap | ratio |
|---|---|---|---|---|---|---|
| constant_fold   | 119 | 48 | 0.0431 | 0.0495 | +0.0064 | **1.15** |
| strength_reduce | 119 | 48 | 0.0200 | 0.0446 | +0.0245 | **2.23** |
| dce             | 119 | 48 | 0.0766 | 0.0325 | -0.0441 | **0.42** |
| cse             | 119 | 48 | 0.0114 | 0.0108 | -0.0006 | **0.94** |
| licm            | 119 | 48 | 0.0123 | 0.0102 | -0.0021 | **0.83** |

Interpretation thresholds from the handoff:
- `ratio ≤ 1.5` — generalizes well; trust the full-corpus fit.
- `1.5 < ratio ≤ 2.0` — mild overfitting; usable but flag in docs.
- `ratio > 2.0` — overfit; the full-corpus RMSE headline does not hold.

### Per-pass verdict

- **constant_fold — generalizes well (ratio 1.15).** Train and test RMSE
  are essentially the same. Full-corpus coefficients are trustworthy.
- **strength_reduce — overfit (ratio 2.23).** Train RMSE 0.0200 jumps to
  test RMSE 0.0446, a 2.23× gap above the handoff's 2.0 overfitting
  threshold. The full-corpus train RMSE of 0.029 reported in section 5
  is a misleading headline number — held-out programs see roughly 2×
  worse error. This corroborates handoff §3A.5 item 8: SR's pass-native
  change-count (e.g. `x*2 → x+x` counts as 1 rewrite) is the noisiest
  proxy for runtime benefit, so a fit that nails the training corpus
  doesn't carry to new programs cleanly.
- **dce — generalizes well (ratio 0.42).** Test RMSE is *lower* than
  train. This is mildly suspicious — it could indicate the test split
  happens to contain DCE-easy programs, or that the train set has high-
  variance DCE outliers that the linear model couldn't fit. Either way
  the absolute test RMSE (0.0325) is comparable to other passes, so the
  generalization claim still holds.
- **cse — generalizes well (ratio 0.94).** Best-behaved pass. The v3
  Phase 2 RMSE of 0.011 holds essentially unchanged on held-out data.
- **licm — generalizes well (ratio 0.83).** Same picture as CSE —
  small train RMSE, smaller test RMSE, robust generalization.

### What changes downstream

The full-corpus coefficient values shipped in
[`linear_cost_model.rs`](../../crates/cjc-cana/src/linear_cost_model.rs)
are **unchanged** — Phase 2 of the training binary still emits them from
the full-corpus fit. The Phase 3 held-out fit is reporting-only and
doesn't alter what the compiler uses today.

But the v3 confidence values (mapped from train RMSE) should be read
with a Phase 3 lens:

- For CF/DCE/CSE/LICM, the v3 confidence (≥0.59) is roughly right; the
  generalization gap is small enough that the held-out picture matches
  the training picture.
- **For SR, the v3 confidence is overstated.** A train RMSE of 0.029
  maps to confidence ≈ 0.77 in v3's `trained_confidence()`, but the
  test RMSE of 0.045 would map to confidence ≈ 0.65. The
  `trained_ranker()` is more bullish on SR than the data justifies.

### What this does *not* tell us

The handoff explicitly lists this as item 4 of 8 critiques (§3A.5).
Items still unverified after this experiment:

- **Item 5 (mean benefits tiny).** Untouched — the held-out RMSE numbers
  are still 1-7× the mean benefit signal. Inherent to the linear model
  + small-benefit regime; needs corpus expansion or non-linear model.
- **Item 6 (`w_alloc_sites = 0`).** Untouched — same corpus, same zero-
  variance feature column. §3A.4 (feature audit) is the experiment for
  this.
- **Item 7 (negative wrong-sign `w_branch_count` for CF/DCE).** Held-out
  RMSE for CF and DCE is fine, so the wrong-sign weights are not catching
  a real generalization bug. They're probably an artifact of column
  correlation in the small corpus, not a load-bearing error. Still worth
  noting in `trained_ranker()`'s doc comment.
- **Item 8 (pass-native ≠ runtime benefit).** §3A.2 (chess RL AB test)
  is the experiment for this. Phase 3 here strengthens the suspicion
  that SR specifically suffers from this mismatch, but doesn't measure
  runtime impact.

### Recommendation

`trained_ranker()` is safe to keep on by default for **CF, DCE, CSE,
LICM** based on this validation. For **SR**, either:

1. Add a doc-comment caveat to `trained_ranker()` flagging SR as having
   ~2× higher real-world error than the training RMSE suggests, OR
2. Reduce SR's emitted confidence (in `trained_base_compile_cost` or the
   `trained_confidence` mapping) to reflect the held-out RMSE rather
   than the training RMSE.

The next validation that would change this picture is §3A.2 (real-
workload AB test on chess RL / PINN / LendingClub). If the AB test
shows trained vs default PassPlans differ but runtime is unchanged for
SR-heavy programs, the held-out finding gets reinforced.
