# ADR-0013: Minimal Package Manager for CJC v1.0

**Status:** Proposed
**Date:** 2026-03-22
**Deciders:** Lead Language Architect, Compiler Pipeline Engineer, Determinism Auditor

## Context

CJC currently has no package or dependency management system. All code must live in a single file or use the incomplete module system (`cjc-module`). As the language approaches v1.0, users need a way to:

1. Share and reuse libraries (e.g., Bastion for solvers, Vizor for visualization)
2. Declare dependencies with version constraints
3. Reproduce builds exactly -- same source + same lockfile = bit-identical output

The package system must respect CJC's core invariant: **deterministic, reproducible execution**. This rules out designs where dependency resolution order, registry fetch timing, or version solving can produce non-deterministic builds.

## Decision

Design a minimal package system with three components: a manifest file (`cjc.toml`), a lockfile (`cjc.lock`), and a resolution algorithm. The initial implementation targets git-based dependencies only; a centralized registry is deferred to a future ADR.

### 1. Manifest Format (`cjc.toml`)

```toml
[package]
name = "my-project"
version = "0.1.0"
edition = "2026"

[dependencies]
bastion = { git = "https://github.com/cjc-lang/bastion.git", tag = "v0.3.0" }
vizor   = { git = "https://github.com/cjc-lang/vizor.git", tag = "v0.2.1" }

[dev-dependencies]
test-utils = { git = "https://github.com/cjc-lang/test-utils.git", tag = "v0.1.0" }

[package.metadata]
deterministic = true   # enforces reproducibility checks on all deps
```

Fields:
- `name`: Package name (kebab-case, alphanumeric + hyphens)
- `version`: Semantic version (major.minor.patch)
- `edition`: Language edition year (governs syntax/semantics)
- `dependencies`: Map of dependency name to source specifier
- `dev-dependencies`: Dependencies only needed for testing

### 2. Dependency Resolution Strategy

Resolution must be **deterministic and reproducible**:

- **Exact version pinning only.** No version ranges in v1. Every dependency specifies an exact tag, branch, or commit hash. This eliminates SAT-solving complexity and non-determinism.
- **Topological sort by name.** When multiple dependencies exist, they are resolved in lexicographic order by package name. This produces a stable ordering regardless of declaration order in `cjc.toml`.
- **No diamond dependency conflicts in v1.** If two packages depend on different versions of the same transitive dependency, resolution fails with a clear error. Conflict resolution (e.g., semver-compatible deduplication) is deferred.
- **Offline-capable.** Once fetched, all dependencies are cached locally. Builds never require network access after the initial fetch.

### 3. Lockfile Format (`cjc.lock`)

```toml
# Auto-generated by `cjc lock`. Do not edit.
# Lockfile version 1

[[package]]
name = "bastion"
version = "0.3.0"
source = "git+https://github.com/cjc-lang/bastion.git?tag=v0.3.0"
commit = "a1b2c3d4e5f6a1b2c3d4e5f6a1b2c3d4e5f6a1b2"
checksum = "sha256:abcdef1234567890abcdef1234567890abcdef1234567890abcdef1234567890"

[[package]]
name = "vizor"
version = "0.2.1"
source = "git+https://github.com/cjc-lang/vizor.git?tag=v0.2.1"
commit = "b2c3d4e5f6a1b2c3d4e5f6a1b2c3d4e5f6a1b2c3"
checksum = "sha256:1234567890abcdef1234567890abcdef1234567890abcdef1234567890abcdef"
```

Fields per package:
- `name`, `version`: Identity
- `source`: Canonical source URL with fetch parameters
- `commit`: Exact git commit hash (40 hex chars) -- ensures reproducibility even if tags move
- `checksum`: SHA-256 of the package source tree -- detects tampering or unexpected changes

The lockfile is checked into version control. `cjc lock` regenerates it; `cjc build` refuses to proceed if the lockfile is stale.

### 4. Registry Options

**v1.0: Git-based only.** Dependencies are fetched directly from git repositories. This avoids the operational complexity of running a centralized registry and aligns with CJC's early-stage ecosystem.

**Future (post-v1.0): Centralized registry.** A `crates.io`-style registry (`pkg.cjc-lang.org`) could be added later. The manifest format already supports extensible source specifiers:

```toml
bastion = "0.3.0"                    # future: shorthand for registry lookup
bastion = { version = "0.3.0" }      # future: explicit registry
bastion = { path = "../bastion" }     # local path dependency
```

### 5. CLI Commands

```bash
cjc init                  # Create cjc.toml in current directory
cjc lock                  # Resolve dependencies, write cjc.lock
cjc fetch                 # Download all locked dependencies to cache
cjc build                 # Compile with dependencies
cjc build --verify-lock   # Fail if lockfile would change (CI mode)
```

### 6. Cache Layout

```
~/.cjc/
  cache/
    git/
      github.com-cjc-lang-bastion-a1b2c3d4/   # content-addressed by commit
      github.com-cjc-lang-vizor-b2c3d4e5/
  registry/                                     # future: registry index
```

## Rationale

- **Exact pinning over semver ranges**: Semver range resolution is inherently non-deterministic across time (new versions change the solution). Exact pinning is simpler, auditable, and aligns with CJC's determinism guarantees.
- **Git-first**: The CJC ecosystem is small. Running a registry is premature. Git repos are universally accessible and already content-addressed.
- **Lockfile with checksums**: The lockfile captures the exact state of every dependency. Checksums detect supply-chain tampering. Together they guarantee that `cjc build` on two different machines with the same lockfile produces identical dependency trees.
- **No version ranges in v1**: This is intentionally restrictive. It eliminates an entire class of reproducibility bugs. Version ranges can be introduced in a future ADR once the ecosystem is large enough to benefit from automatic updates.

## Consequences

**Positive:**
- Builds are fully reproducible: same `cjc.toml` + `cjc.lock` = identical dependency tree.
- No registry infrastructure needed for v1.0 launch.
- Simple mental model: every dependency is pinned to an exact commit.
- Offline builds work after initial fetch.

**Negative:**
- Users must manually update dependency versions (no `cjc update` with range resolution in v1).
- Diamond dependency conflicts produce hard errors rather than automatic resolution.
- Git-only means no private registry support in v1 (private git repos work, but require SSH/token auth).

**Known limitations:**
- No support for binary/pre-compiled packages -- all dependencies are source-only.
- No build scripts or pre/post-install hooks (intentional: avoids non-deterministic build steps).
- No workspace/monorepo support in v1.

## Implementation Notes

- Crates affected: `cjc-cli` (new subcommands), new crate `cjc-pkg` (resolution, fetching, lockfile)
- The `cjc-module` crate must be extended to resolve imports from fetched packages in the cache
- Lockfile parsing can reuse the existing TOML parser or add a minimal one (zero-dep constraint)
- Determinism auditor must verify that resolution produces identical lockfiles across platforms
- See also: ADR-0004 (SplitMix64 determinism philosophy applies to build reproducibility)
